# 7 Reinforcement learning: From policy gradients to GRPO

### This chapter covers
* RL for LLMs: PPO, rewards, baselines, GRPO, R1 models
* Why PPO needs policy, value, reward, and reference models
* PyTorch GRPO walkthrough with verifier rewards and training steps

In the previous chapters, we built the architectural backbone of a DeepSeek-style model from scratch. We implemented Multi-Head Latent Attention, Decoupled RoPE, Mixture-of-Experts routing, Multi-Token Prediction, and the training pipeline that turns those components into a capable base model. That base model can predict text, but prediction alone is not the same as deliberate problem solving.

This chapter moves into post-training: the stage where a pretrained language model is pushed toward behaviors that are useful for reasoning. The central tool is reinforcement learning. Instead of showing the model only the next token in a dataset, reinforcement learning lets the model try complete responses and then assigns a reward to those responses. The training update then makes rewarded behavior more likely in future generations.

The DeepSeek-R1 result is important because it showed that a large base model could develop long-form reasoning behavior when trained with Group Relative Policy Optimization, or GRPO, and verifiable rewards. The model was not simply memorizing human-written chain-of-thought examples. It was rewarded for solving objectively checkable tasks, and under that pressure, it began to produce longer, more self-corrective solutions.

We will build this code carefully. First, we will map ordinary reinforcement learning concepts to language generation. Then we will derive the basic policy-gradient update and see how PPO became a practical stabilizer. From there, we will move to GRPO, the simplification used in DeepSeekMath and then in DeepSeek-R1-Zero. The final section gives concrete PyTorch excerpts so that the next chapter can truthfully build on an implementation path, not only on theory.

## 7.1 The reinforcement learning framework

Reinforcement learning is a framework for learning from interaction. An agent observes the current state of an environment, takes an action, receives a reward, and then observes a new state. Over many interactions, the agent learns which actions tend to produce higher rewards.

This is different from supervised learning. In supervised learning, each example normally comes with a target answer. The model sees an input and learns to imitate the target. Reinforcement learning is weaker and more flexible: the agent may not be told the exact correct action at every step. It may only discover whether its overall behavior was useful after several actions.

That delayed feedback is the main reason reinforcement learning is useful for reasoning. When a language model solves a math problem, we often care about the final answer. We might not know the exact sequence of intermediate tokens the model should generate. Reinforcement learning can reward a correct final answer and let the model discover which intermediate behaviors help it get there.

### 7.1.1 The agent-environment interface

We will use one concrete example throughout this section: a mobile cleaning robot. The robot is the agent. The room is the environment. At each step, the robot observes the room: where dirt remains, where furniture blocks movement, whether the battery is low, and where the charging dock is located.

The robot then chooses an action. It may move forward, turn left, vacuum the current tile, or return to the dock. After the action, the environment changes. A tile may become clean, the battery may decrease, or the robot may bump into an obstacle. The environment returns a reward signal. Cleaning dirt might produce a positive reward. Wasting battery or colliding with a wall might produce a negative reward.

The important feature is the loop. The robot does not receive a single static example. It acts, observes the consequences, and acts again. A policy is the rule the agent uses to choose actions from states. In neural-network reinforcement learning, the policy is represented by model parameters. Training changes those parameters so that actions leading to higher long-term reward become more likely.

*Figure 7.2 The agent-environment interface. The cleaning robot observes the room, chooses an action, receives a reward, and then repeats the cycle from the updated room state.*

This example is deliberately mundane. We do not need many unrelated analogies. The same four objects are enough: state, action, reward, and policy. The rest of the chapter will translate those objects into language-model terms.

### 7.1.2 Mapping reinforcement learning to language generation

At first glance, a language model does not look like a cleaning robot. It does not move through a room. It predicts tokens. But token generation is still a sequential decision process.

The state at time t is the prompt given by the user plus all tokens the model has generated so far in the current response. The action is the next sampled token. The transition is simple: append that token to the context. The new state is the old state plus the sampled token. When the model finishes the response, a reward function scores the response.

Consider the prompt: Roger has 5 tennis balls. He buys 2 more cans of tennis balls. Each can has 3 tennis balls. How many tennis balls does Roger have now? The correct answer is 11. At the first generation step, the state is only the prompt. The model samples a first token, perhaps 'Roger' or 'He'. At the next step, the state is the prompt plus that first token. This continues until the model produces a complete answer.

The word sampled matters. During ordinary inference, we might use greedy decoding or low-temperature sampling. During reinforcement-learning rollouts, however, sampling is central. The model samples from its probability distribution so that training can explore different responses for the same prompt. If the model always chose the single most likely token, it would be much harder to discover alternative solution paths.

*Figure 7.3 Mapping reinforcement learning concepts to LLM generation. The growing context is the state, each sampled next token is an action, and the final response receives a reward.*

### 7.1.3 Sparse rewards and credit assignment

Language-model reinforcement learning usually uses sparse rewards. A reward is sparse when it is available only after a sequence of actions. In the Roger problem, the verifier can score the final answer, but it does not necessarily score every intermediate token. The reward might be 1 if the final answer is 11 and 0 otherwise.

Sparse rewards create a credit-assignment problem. If the model gets the answer wrong, which token caused the error? Was the arithmetic wrong, did the model misread the question, or did it format the answer badly? The reward does not directly say. Policy-gradient methods solve this by updating the probabilities of all sampled actions in proportion to the outcome. Later refinements, such as baselines and advantages, make this update less noisy.

Negative learning signals can also appear even when raw verifier rewards are only 0 and 1. If a completion receives a reward of 0 while the baseline for that prompt is 0.5, its advantage is negative. The model is then pushed away from that sampled behavior. In other settings, reward functions may also assign explicit negative values for invalid output, unsafe output, or severe format violations. In this chapter, we will keep the verifier examples simple but still explain where negative advantages come from.

## 7.2 Policy-gradient methods: Updating the LLM with rewards

The core question is now: how do we update a language model from a reward? The model produces a probability distribution over the vocabulary at each step. During a rollout, it samples a token from that distribution. After the response receives a reward, we want to increase the probability of sampled tokens that helped and decrease the probability of sampled tokens that hurt.

This is the job of a policy-gradient method. A policy-gradient method estimates how changing the policy parameters would change expected reward. In language-model terms, the policy parameters are the weights of the LLM. The policy is the model's probability distribution over the next token.

![Title: dimensions:[378x211] - Description: 075.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/07__image005.png)
*Figure 7.5 A probability distribution over the next token. During RL training, the model samples from this distribution rather than always selecting the highest-probability token.*

## 7.3 Sampling actions instead of taking argmax

Suppose the model is deciding the next token after a partial answer. It might assign probability 0.42 to 'has', 0.23 to 'now', 0.11 to 'will', and smaller probabilities to other tokens. During a rollout, the token 'has' is most likely to be sampled, but it is not guaranteed. A lower-probability token can still be selected.

This stochasticity is not an implementation detail; it is required for exploration. If the model never samples lower-probability actions, it never learns whether those actions lead to better outcomes. Policy-gradient training relies on the log probability of the action that was actually sampled. The update asks: Should the model assign a higher or lower probability to this sampled action in this sampled state?

For LLMs, this repeats across every generated token. A full response is a trajectory: a sequence of states and sampled actions. The reward is attached to the trajectory, and the update changes token probabilities along that trajectory.

### 7.3.1 The basic policy-gradient update

A simplified policy-gradient update can be written as follows:

> θₜ₊₁ = θₜ + α ∑ₖ ∇θ log πθ(aₖ | sₖ) R

Here θ represents the model parameters, α is the learning rate, sₖ is the state at token step k, and aₖ is the token sampled at that step. The symbol πθ is the policy: the LLM's probability distribution over the next token under parameters θ. The term πθ(aₖ | sₖ) is the probability that the model assigns to the sampled token aₖ given the current state sₖ.

The gradient of the log probability tells us how to change the model parameters so the sampled action becomes more likely. Multiplying by reward R controls the direction and strength of the update. If R is high, the sampled actions are reinforced. If R is low relative to a baseline, the sampled actions are discouraged.

This equation is intentionally compact. A real LLM implementation must handle sequence masks, variable response lengths, old-policy log probabilities, KL penalties, and batched computation. But the idea is already present: reward becomes a learning signal for the tokens the model actually sampled.

### 7.3.2 Why raw rewards are too noisy

Raw reward is a blunt instrument. Imagine two prompts. One is easy, and most completions get a reward of 1. The other is hard, and most completions get a reward of 0. A completion with reward 1 on the easy prompt may not be impressive; a completion with reward 0 on the hard prompt may still be better than most alternatives. Raw reward alone does not tell us how good a response is relative to what was expected for that prompt.

This is why reinforcement-learning algorithms usually use an advantage instead of the raw reward. An advantage measures how much better or worse an outcome was compared with a baseline. If a response is better than expected, the advantage is positive. If it is worse than expected, the advantage is negative.

> Aₖ = Rₖ - baseline(sₖ)

The baseline should not change which actions are optimal, but it can dramatically reduce variance. Lower variance means the update is less noisy and training is more stable. Proximal Policy Optimization (PPO) and GRPO differ mainly in how they obtain this baseline.

## 7.4 PPO: The standard practical baseline

Proximal Policy Optimization, or PPO, became a standard reinforcement-learning algorithm because it stabilizes policy updates while remaining easier to implement than older trust-region methods. PPO was introduced by Schulman et al. in Proximal Policy Optimization Algorithms, https://arxiv.org/abs/1707.06347. It builds on the trust-region idea from Schulman et al., Trust Region Policy Optimization, https://arxiv.org/abs/1502.05477.

The trust-region intuition is simple: do not let one update move the policy too far from the policy that produced the current batch of data. If the model changes too much after a small batch of rollouts, the training signal can become unreliable. The new policy may assign very different probabilities to the same sampled tokens, causing unstable updates.

PPO controls this with a probability ratio. The ratio compares the probability of a sampled token under the new policy with its probability under the old policy that produced the rollout.

> ratioₖ(θ) = πθ(aₖ | sₖ) / πold(aₖ | sₖ)

If the ratio is close to 1, the new policy is close to the old policy for that sampled action. If the ratio is much larger than 1, the new policy is making the action far more likely. If it is much smaller than 1, the new policy is making the action far less likely. PPO clips this ratio to prevent overly large updates.

### 7.4.1 PPO components for LLM training

In language-model post-training, PPO commonly involves four components. First, there is the trainable policy model, the LLM being updated. Second, there is a reward signal, often produced by a learned reward model or by a verifier. Third, there is a value model that estimates expected future reward and provides the baseline for advantage calculation. Fourth, there is a frozen reference model used for KL control.

The reference model is usually a frozen copy of the model before RL training. It gives us a stable anchor. If the trained policy moves too far away from the reference policy, the model may exploit the reward signal, degrade language quality, or drift away from useful instruction-following behavior. A KL penalty discourages that drift.

The value model is different from the reference model. It is not a frozen anchor. It is trained to predict expected reward from partial responses. In many implementations, it is a value head attached to the policy backbone, but conceptually it is a critic: it estimates how promising a state is. PPO uses that estimate to compute advantages.

*Figure 7.6 PPO-style language-model training uses a trainable policy, a reward signal, a learned value baseline, and a frozen reference policy for KL control.*

### 7.4.2 Where the value function comes from

The value function Vϕ(sₖ) is not magic. It is a learned predictor. During rollouts, the training system observes partial states and eventual returns. The value model is trained to predict the expected return from each state. If the final reward for a response is high, states along that response become examples of states that led to good outcomes. If the final reward is low, they become examples of states that led to poor outcomes.

> Aₖ = Rₖ - Vϕ(sₖ)

Here ϕ denotes the parameters of the value model. The notation is separate from θ because the value model and policy are conceptually different networks, even when they share some layers in an implementation. The policy chooses tokens; the value model predicts how good the current partial response is expected to be.

This value model can be useful, but it is expensive. It adds memory, forward passes, training loss terms, and engineering complexity. For a very large language model, even an extra head or critic path can matter. GRPO was designed to remove this learned value-function model while preserving a useful baseline.

### 7.4.3 Why the reference model matters

The reference model exists because reward functions are imperfect. If a model is rewarded only for final math correctness, it might learn strange formatting or overly long responses. If a reward model is trained from preference data, the policy may exploit quirks of the reward model. The reference policy provides a constraint: improve on the task, but do not move too far from the pretrained model's distribution.

In PPO-style LLM training, the KL term is often computed token by token. The system compares the trained policy's log probability for generated tokens with the reference policy's log probability for those same tokens. Large divergence creates a penalty. This does not guarantee safety or correctness, but it is a practical stabilizer.

When we discuss GRPO, keep the reference model in the picture. GRPO removes the value model. It does not remove the reference model. The reference-policy KL penalty remains important in DeepSeek-style training.

## 7.5 GRPO: DeepSeek's value model simplification

Group Relative Policy Optimization, or GRPO, was introduced in DeepSeekMath: Pushing the Limits of Mathematical Reasoning in Open Language Models, https://arxiv.org/abs/2402.03300. DeepSeek later used the same family of ideas in the DeepSeek-R1 training recipe. The key simplification is that GRPO removes the learned value model.

Instead of training a value model to estimate a baseline, GRPO samples a group of completions for the same prompt. Each completion receives a reward. The rewards in that group are then normalized. A completion is good if it is better than the other completions sampled for the same prompt; it is bad if it is worse than the group.

This makes the baseline prompt-specific without requiring a critic. For the Roger prompt, suppose the model samples four completions. Two end with 11 and receive a reward of 1. Two end with incorrect answers and receive a reward of 0. The group mean is 0.5. Correct completions receive positive advantages, and incorrect completions receive negative advantages.

![Title: dimensions:[378x211] - Description: 077.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/07__image007.png)
*Figure 7.7 GRPO group sampling with reference control. Multiple completions are generated for the same prompt, scored, normalized into advantages, and updated under a KL constraint.*

### 7.5.1 Group-relative advantages

The group-relative advantage for completion i can be written as:

> Aᵢ = (rᵢ - mean(r₁, …, rG)) / (std(r₁, …, rG) + ε)

Here, G is the number of completions sampled for the same prompt, rᵢ is the reward for completion i, and ε is a small constant for numerical stability. If all completions receive the same reward, the standard deviation is near zero, so the implementation must avoid division by zero.

This equation explains the negative advantages that appeared earlier. The verifier may only assign 0 or 1, but after subtracting the group mean, a reward of 0 can become negative. That negative advantage tells the optimizer to reduce the probability of the sampled trajectory relative to better completions for the same prompt.

![Title: dimensions:[378x211] - Description: 078.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/07__image008.png)
*Figure 7.8 Group-relative advantage calculation. For binary rewards [1, 1, 0, 0], the group mean is 0.5, so correct completions receive positive advantages and incorrect completions receive negative advantages.*

Per-token advantages are more nuanced than this diagram can show. The same completion-level advantage is often broadcast over the generated tokens, while masks ensure that prompt tokens and padding tokens do not contribute to the loss. Production implementations also account for response length, KL penalties, and token-level log probabilities.

### 7.5.2 The full GRPO objective

It is tempting to write GRPO as a simple policy-gradient update with a group-relative advantage. That is useful for intuition, but it is not the full objective. The full GRPO-style loss keeps PPO's clipped ratio and adds a KL term against the reference policy.

> LGRPO(θ) = E[min(ratioₖ(θ)Aᵢ, clip(ratioₖ(θ), 1-ε, 1+ε)Aᵢ) - β KL(πθ || πref)]

The ratio term compares the current policy with the old policy that generated the sampled completions. The clipping term limits update size. The KL term discourages drift from the frozen reference policy. The group-relative advantage replaces the value-model advantage; it does not replace the stabilizers.

When this objective is implemented as a minimization loss in PyTorch, we usually negate it. Optimizers minimize by default, so maximizing the objective becomes minimizing the negative objective.

### 7.5.3 Why GRPO is cheaper than PPO

GRPO is not cheaper because it samples fewer completions. In fact, it explicitly samples multiple completions per prompt. The fair comparison is at the same total number of sampled completions. Under that comparison, GRPO is cheaper because it eliminates the learned value model and its training path.

| Component | PPO-style training | GRPO-style training |
|---|---|---|
| Policy model | Trainable LLM | Trainable LLM |
| Reward signal | Reward model or verifier | Reward model or verifier |
| Value model | Required for advantage baseline | Removed; group statistics provide the baseline |
| Reference model | Frozen model for KL control | Frozen model for KL control |
| Main saving | None from the critic path | No value-model forward/backward path |

This distinction matters for readers. A single PPO example with one sampled response is pedagogically simple, but actual PPO training also uses many samples. GRPO's advantage is not that it avoids sampling. It is that it avoids learning a separate value function.

## 7.6 Reinforcement learning with verifiable rewards

The next simplification is reinforcement learning with verifiable rewards, or RLVR. In RLVR, the reward is produced by a deterministic checker rather than a learned reward model. For a math problem, the checker can compare the final answer with the known answer. For code, the checker can run unit tests or a compiler. For symbolic tasks, the checker can validate a result against a formal rule.

This is powerful because the reward is less subjective. If the prompt asks for the answer to an arithmetic problem, a correct final number can be checked automatically. If a code problem comes with tests, passing tests can be scored automatically. The model can generate many candidate solutions and receive a reliable signal without human labels for every response.

RLVR does not solve every alignment problem. It applies only when correctness is objectively verifiable. Open-ended dialog helpfulness, creative writing quality, nuanced reasoning, harmlessness, and preference-sensitive assistant behavior do not have deterministic checkers. For those tasks, learned reward models, preference data, human evaluation, or other alignment signals remain necessary.

![Title: dimensions:[378x211] - Description: 0710.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/07__image010.png)
*Figure 7.10 RLVR can replace learned reward models for objectively verifiable tasks such as math and code; open-ended assistant behavior still requires preference signals.*

### 7.6.1 Designing a verifier

A verifier should be strict enough to reward real correctness and flexible enough not to reject harmless formatting differences. For a toy math task, we can extract the final number from the completion and compare it with the gold answer. For a production math verifier, we may need symbolic equivalence, unit normalization, tolerance for decimals, or structured answer extraction.

The reward function also shapes behavior. If the verifier only checks the last number, the model may learn to produce long, messy reasoning followed by the right number. If the verifier also checks the format, the model may learn to separate reasoning from the final answer. If the verifier is too brittle, the model may be punished for correct answers written in a different form.

DeepSeek-R1-Zero used rule-based rewards for reasoning tasks, including correctness rewards and format rewards. The format reward encouraged responses to follow the expected reasoning/final-answer structure. The correctness reward checked whether the final answer was right for tasks where that could be verified.

### 7.6.2 Avoiding overstatement

It is important not to overstate RLVR. The lesson is not that reward models are obsolete. The lesson is that reward models are not needed when a high-quality deterministic verifier exists. Math and code are natural targets because many tasks have objective answers. General assistant behavior is different.

DeepSeek-R1 itself reflects this distinction. Its later training stages use a hybrid reward design: rule-based rewards for verifiable reasoning tasks and learned reward models for tasks that require preference judgments, such as helpfulness and harmlessness. This is the practical recipe: use verifiers where they are reliable, and use preference signals where the target behavior is not mechanically checkable.

## 7.7 How reasoning emerges in DeepSeek-R1-Zero

DeepSeek-R1-Zero is the cleanest experiment in the R1 family. It starts from DeepSeek-V3-Base and applies large-scale reinforcement learning directly, without first training on supervised chain-of-thought examples. The reward is focused on verifiable reasoning tasks. The model is not told to imitate a human-written solution trace; it is rewarded for solving the task.

During training, the model begins to allocate more tokens to difficult problems. Longer responses are not valuable by themselves, but they give the model room to try an approach, check intermediate results, notice mistakes, and revise. The DeepSeek-R1 report highlights an 'aha moment' in which the model explicitly notices that it should re-evaluate its approach.

The `<think>` tag in the report marks the reasoning region of the model's response. It is not a Python tag or a special interpreter construct. It is a formatting convention used to separate the model's internal reasoning-style text from the final answer region. In public deployments, reasoning text may be hidden or summarized, but in the report, it is shown to illustrate the behavior learned during RL.

![Title: dimensions:[378x205] - Description: 0711.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/07__image011.png)
*Figure 7.11 The aha moment example from DeepSeek-R1 training. The `<think>` tag marks the model's reasoning region; the notable behavior is that the model notices a flawed path and changes strategy.*

This result should be interpreted carefully. R1-Zero shows that reasoning-like behavior can emerge from RL pressure on verifiable tasks. It does not mean that every reasoning behavior is guaranteed to emerge from any reward. It also does not mean the resulting model is polished. The report notes problems such as poor readability and language mixing. These issues motivate the more engineered DeepSeek-R1 pipeline.

## 7.8 DeepSeek-R1, R1-Zero, and Distill

The DeepSeek-R1 family is easy to misunderstand because the names are similar. DeepSeek-R1-Zero, DeepSeek-R1, and DeepSeek-R1-Distill are related, but they are not the same artifact.

DeepSeek-R1-Zero is the pure RL experiment. It asks whether a strong base model can develop reasoning behavior through RL with verifiable rewards, without supervised chain-of-thought data. DeepSeek-R1 is the more usable final model, produced by a staged pipeline that combines cold-start supervised fine-tuning, reasoning-oriented RL, rejection sampling, mixed-data supervised fine-tuning, and final RL. DeepSeek-R1-Distill models are smaller student models trained on data generated by DeepSeek-R1.

![Title: dimensions:[378x211] - Description: 0713.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/07__image013.png)
*Figure 7.13 The DeepSeek-R1 model lineup. R1-Zero demonstrates emergent reasoning; R1 improves readability and robustness; Distill transfers R1-generated reasoning data into smaller students.*

### 7.8.1 From R1-Zero to R1

The arrow from R1-Zero to R1 is a data flow, not a parameter flow. R1-Zero's outputs are sampled, filtered, and used as training data. R1 is not simply R1-Zero trained longer. It is a staged system built around the insight that RL can generate useful reasoning traces, but that those traces need filtering and refinement before they become a polished assistant model.

The first stage of DeepSeek-R1 uses cold-start supervised data. DeepSeek creates a small set of long chain-of-thought examples with a more readable style and fine-tunes the base model on them. This gives the model a better initial reasoning format before heavy RL.

The next stage uses reasoning-oriented RL with rule-based rewards. This stage resembles the R1-Zero experiment but starts from a better-formatted model. It also includes a language-consistency reward to reduce the language mixing observed in R1-Zero.

After that, DeepSeek uses rejection sampling and supervised fine-tuning. The system samples many outputs, filters them for correctness and readability, and mixes reasoning data with broader non-reasoning instruction data. This step restores general assistant behavior that pure verifier-based RL does not directly optimize.

The final RL stage uses a hybrid reward. For verifiable tasks, rule-based rewards still apply. For open-ended helpfulness and harmlessness tasks, learned reward models provide the preference signal. This is why the R1 pipeline should not be summarized as 'no reward model.' It uses verifiers where possible and reward models where necessary.

*Figure 7.14 DeepSeek-R1 training pipeline, adapted from Guo et al., DeepSeek-R1, Figure 2. The flow from R1-Zero into R1 represents filtered training data, not direct parameter transfer.*

### 7.8.2 Why cold-start data matters

The cold-start stage in DeepSeek-R1 is easy to skip over, but it solves a practical problem. Pure RL can discover useful behavior, but it does not guarantee that the behavior is pleasant to read, consistently formatted, or aligned with the conventions expected from an assistant. R1-Zero demonstrated reasoning behavior, but the report also described readability problems and language mixing.

Cold-start supervised examples give the model a better initial response style before the main reasoning RL stage. Instead of asking RL to discover both correctness and readable formatting from scratch, the model begins with a small amount of curated reasoning data. This is a common pattern in post-training: use supervised learning to set the initial behavior distribution, then use RL to improve outcomes under a reward.

For our purposes, the lesson is that supervised fine-tuning and reinforcement learning are not competing tools. They solve different parts of the problem. Supervised fine-tuning is good at teaching format, style, and imitation of known examples. Reinforcement learning is good at optimizing behavior when we can score outcomes. DeepSeek-R1 uses both.

A small cold-start dataset can also reduce wasted exploration. If the model already knows how to produce a structured solution with a final answer region, the verifier has an easier job, and the RL signal is cleaner. Without that structure, the model may spend many rollouts producing outputs that are hard to parse or that fail format requirements despite containing useful reasoning.

### 7.8.3 Rejection sampling as data construction

Rejection sampling appears in the R1 pipeline after the reasoning-oriented RL stage. The idea is straightforward: sample many candidate responses from a strong model, keep the ones that satisfy correctness and quality criteria, and discard the rest. The accepted responses become supervised training data.

This is not reinforcement learning by itself. It is a data-generation and filtering step. But it depends on the RL-trained model because that model can produce stronger reasoning traces than the original base model. The pipeline uses RL to improve the generator, then uses filtering to convert the generator's best outputs into a supervised dataset.

The filtering criteria matter. For verifiable reasoning tasks, correctness can be checked by a rule-based verifier. For readability and general instruction-following data, the filter may involve heuristics, model-based judgments, or human preference data. The goal is to avoid training the next stage on low-quality or inconsistent outputs.

This is another reason the flow from R1-Zero to R1 should be described as data flow. The model does not simply copy weights from the pure-RL experiment into the final product. Instead, strong outputs are harvested, filtered, and mixed with other data sources to train a more balanced checkpoint.

### 7.8.4 Restoring general assistant behavior

A model optimized heavily on math and code verifiers may become better at verifiable reasoning while becoming less balanced as a general assistant. It might overuse long reasoning, respond awkwardly to simple requests, or focus on answer formats that make sense for benchmark tasks but not for everyday interaction.

DeepSeek-R1 addresses this with mixed supervised data after rejection sampling. The training data includes reasoning examples, but it also includes non-reasoning instruction data such as writing, editing, factual question answering, role play, and other general assistant tasks. This stage helps preserve the broad usefulness of the model.

This design is important for readers building their own systems. If you train only on verifiable math rewards, you should not expect the model to become a polished general assistant. The reward defines the pressure. A narrow reward creates narrow improvement unless it is combined with broader data and alignment signals.

In a book project like ours, this distinction prevents an overly simple story. GRPO plus RLVR is the key algorithmic idea, but the final R1 system is a pipeline. The pipeline combines data curation, supervised learning, reinforcement learning, rejection sampling, and hybrid rewards.

### 7.8.5 The final hybrid reward stage

The last RL stage in DeepSeek-R1 uses a hybrid reward because the model must handle both verifiable and non-verifiable tasks. For a math problem, a rule-based checker can score correctness. For a helpfulness prompt, there may be no single objectively correct response. A learned reward model or preference-based signal is more appropriate.

This hybrid design is the practical answer to the question: Does RLVR replace reward modeling? No. RLVR replaces reward modeling for tasks where a reliable verifier exists. Reward modeling remains useful when the desired behavior is a matter of preference, nuance, safety, or style.

The final R1 model inherits strengths from multiple sources. Verifiable rewards strengthen reasoning on tasks with objective answers. Supervised data improves format and general usefulness. Preference-oriented rewards align responses with human judgments on open-ended tasks. Distillation then transfers many of these behaviors into smaller models.

The main engineering principle is to match the reward source to the task. Use exact checkers where exact checking is possible. Use learned preference signals where the target cannot be reduced to a deterministic answer.

### 7.8.6 Distillation into smaller models

DeepSeek-R1-Distill models answer a practical deployment question: how can the reasoning behavior of a large model be transferred into smaller models? The distilled models are not trained by repeating the full large-scale RL pipeline on small models. Instead, smaller open-source base models, including Qwen and Llama variants, are fine-tuned on a large supervised dataset generated by DeepSeek-R1.

This matters because pure RL can be expensive and unstable, especially for smaller models that may not have the same base capabilities as the teacher. Distillation packages the teacher's solution patterns into a supervised dataset. The student learns to imitate successful reasoning traces without needing to rediscover them through expensive exploration.

The DeepSeek-R1 report found that distillation can be very effective. A distilled, smaller model can outperform a similarly sized model trained with RL alone on several reasoning benchmarks. The practical lesson is that RL can discover or strengthen reasoning behavior in a large, capable model, while distillation can transfer that behavior into cheaper models.

Distillation also changes the training economics. RL requires repeated sampling, scoring, and policy updates. Supervised fine-tuning on distilled data is simpler and more predictable. Once a strong teacher has generated a large dataset of high-quality reasoning traces, many student models can be trained from that dataset without running the full RL pipeline again.

This does not mean distillation is always better than RL. Distillation can transfer behavior that the teacher already has, but it does not usually discover new behavior beyond the teacher. RL can push the teacher into new regions of behavior when the reward is reliable. The two methods complement each other: RL improves the teacher; distillation distributes the teacher's behavior.

### 7.9.1 A minimal verifier

The first piece is a deterministic verifier. For an answer-only math prompt, a toy verifier can extract the final number from the completion and compare it with the gold answer.

##### Listing 7.2 A minimal verifier for answer-only math prompts
```python
import re
def extract_final_number(text: str) -> str | None:
    """Return the last integer or decimal number found in a completion."""
    matches = re.findall(r"-?\d+(?:\.\d+)?", text)
    return matches[-1] if matches else None
def verify_math_answer(completion: str, gold: str) -> float:
    """A toy RLVR reward for answer-only math tasks."""
    predicted = extract_final_number(completion)
    if predicted is None:
        return 0.0
    return 1.0 if predicted == str(gold) else 0.0
```

This verifier is intentionally simple. It is good enough to illustrate the RLVR mechanism, but it is not robust enough for a production math training run. A stronger verifier would parse structured final answers, handle equivalent fractions and decimals, and reject completions that put multiple conflicting answers in the output.

### 7.9.2 Computing group-relative advantages

The next piece is GRPO's group-relative baseline. Suppose we have a batch of prompts, and for each prompt, we sampled G completions. The reward tensor has shape `[batch_size, group_size]`. We normalize across the group dimension.

##### Listing 7.3 Computing group-relative advantages
```python
import torch
def group_advantages(rewards: torch.Tensor, eps: float = 1e-6) -> torch.Tensor:
    """Normalize rewards within each prompt group.
 
    Args:
        rewards: Tensor of shape [batch_size, group_size].
        eps: Minimum standard deviation for numerical stability.
 
    Returns:
        Tensor of normalized advantages with the same shape as rewards.
    """
    mean = rewards.mean(dim=1, keepdim=True)
    std = rewards.std(dim=1, keepdim=True).clamp_min(eps)
    return (rewards - mean) / std
```

For rewards [1, 0, 1, 0], the mean is 0.5. The correct completions receive positive normalized advantages, and the incorrect completions receive negative normalized advantages. This is how binary verifier rewards become reinforcement and penalization signals.

### 7.9.3 The clipped GRPO loss

The loss function brings together the current policy, the old policy that generated the completions, the frozen reference policy, and the advantages. In an actual language-model trainer, log probabilities are computed per generated token. The old log probabilities are stored during rollout. The reference log probabilities are computed with the frozen reference model.

##### Listing 7.4 The clipped GRPO loss with reference-policy KL
```python
def grpo_loss(
    logp: torch.Tensor,
    old_logp: torch.Tensor,
    ref_logp: torch.Tensor,
    advantages: torch.Tensor,
    eps: float = 0.2,
    beta: float = 0.04,
) -> torch.Tensor:
    """Compute the clipped GRPO objective as a minimization loss.
 
    Shapes:
        logp, old_logp, ref_logp: [batch_size, group_size, tokens]
        advantages: [batch_size, group_size]
    """
    ratio = torch.exp(logp - old_logp)
    clipped = ratio.clamp(1.0 - eps, 1.0 + eps)
    adv = advantages.unsqueeze(-1)
    surrogate = torch.minimum(ratio * adv, clipped * adv)
    kl = logp - ref_logp
    return -(surrogate - beta * kl).mean()
```

This listing is still simplified. For clarity, it omits attention masks and response masks. In a real trainer, prompt tokens should not contribute to the policy loss, and padding tokens must be ignored. The KL estimate may also use a more carefully chosen formulation. The listing is meant to expose the structure, not to replace a production RL library.

### 7.9.4 One training-step skeleton

The training step wires the pieces together. It samples a group of completions, verifies them, computes group-relative advantages, scores the completions under the current and reference policies, computes the GRPO loss, and applies an optimizer step.

##### Listing 7.5 One GRPO + RLVR training-step skeleton
```python
def train_step(batch, policy, reference, optimizer, group_size: int = 4):
    """One GRPO + RLVR training step skeleton."""
    prompts, gold_answers = batch
    completions, old_logp = sample_group(policy, prompts, group_size)
 
    rewards = torch.tensor(
        [
            [verify_math_answer(y, gold) for y in group]
            for group, gold in zip(completions, gold_answers)
        ],
        device=old_logp.device,
    )
    advantages = group_advantages(rewards)
 
    logp = sequence_logprobs(policy, prompts, completions)
    with torch.no_grad():
        ref_logp = sequence_logprobs(reference, prompts, completions)
 
    loss = grpo_loss(logp, old_logp, ref_logp, advantages)
    optimizer.zero_grad(set_to_none=True)
    loss.backward()
    optimizer.step()
    return {"loss": float(loss.detach()), "reward": float(rewards.mean())}
```

The placeholders `sample_group` and `sequence_logprobs` are where a real LLM implementation would connect to a tokenizer and `model.generate`-style rollout code. `sample_group` must return completions and the old-policy token log probabilities. `sequence_logprobs` must score those same generated tokens under the current policy and the frozen reference model.

## 7.10 A worked GRPO example

Before leaving the algorithm, let us walk through one complete GRPO update in words. This example uses the same Roger tennis-ball prompt because the arithmetic is simple enough that the reinforcement-learning mechanics stay visible.

The prompt is: Roger has 5 tennis balls. He buys 2 more cans of tennis balls. Each can has 3 tennis balls. How many tennis balls does Roger have now? The correct calculation is 5 + 2 × 3 = 11. We ask the current policy to sample four completions for this same prompt.

| Completion | Final answer | Verifier reward |
|---|---|---|
| A | 11 | 1 |
| B | 10 | 0 |
| C | 11 | 1 |
| D | 16 | 0 |

The raw rewards are [1, 0, 1, 0]. The group mean is 0.5. The group standard deviation is approximately 0.577 if we use the unbiased PyTorch default for a group of four values, or 0.5 if we use the population definition. The exact scaling is less important than the sign and relative magnitude. The two correct completions are above the group mean; the two incorrect completions are below it.

After normalization, completions A and C receive positive advantages. Completions B and D receive negative advantages. The policy update, therefore, increases the probabilities of the sampled token sequences in A and C and decreases the probability of the sampled token sequences in B and D, subject to clipping and KL control.

Notice what the algorithm does not know. It does not know that completion B made an addition mistake while completion D counted all cans incorrectly. It only knows that their final answer failed the verifier. The model's prior knowledge and the distribution of many future samples provide the pressure that gradually favors reasoning paths that lead to correct answers.

This is why large-scale RLVR needs many prompts. A single prompt cannot teach arithmetic. But across thousands or millions of verifiable tasks, the model repeatedly sees that certain reasoning patterns lead to reward and others do not. Over time, the policy becomes more likely to produce productive intermediate steps, even though the reward is often attached only to the final answer.

### 7.10.1 What happens to each token?

A common source of confusion is whether the algorithm rewards individual words or the whole answer. The verifier normally scores the whole answer. But the policy-gradient loss is computed from token log probabilities. This means the same completion-level advantage is applied across the generated tokens in that completion.

If completion A receives a positive advantage, the generated tokens in A are pushed up together. If completion B receives a negative advantage, its generated tokens are pushed down together. This is an imperfect credit-assignment signal, but it becomes useful statistically. Tokens and patterns that appear frequently in successful completions become more likely. Tokens and patterns that appear frequently in failed completions become less likely.

The KL term complicates this slightly. A token that receives a positive advantage may still be constrained if increasing its probability would move the policy too far from the reference model. Similarly, the clipping term prevents one batch from making an excessively large change. The final update is the result of advantage pressure, clipping, and reference-policy pressure acting together.

This explains why the simplified equation and the full objective must be kept separate. The simplified policy-gradient equation teaches the direction: reward-weighted log probabilities. The GRPO objective teaches the actual training rule: group-relative advantages with clipping and reference-policy KL.

### 7.10.2 Why multiple samples for the same prompt matter

GRPO's group structure is not just a batching trick. Sampling multiple completions for the same prompt creates a local comparison. The model is not merely told that reward 1 is good and reward 0 is bad. It is told which completions were better or worse for the same input.

This local comparison is especially useful when prompts differ in difficulty. If a prompt is easy, many completions may be correct. If a prompt is hard, most completions may be wrong. A global baseline would mix these cases together. A group-relative baseline compares responses within the same prompt, which makes the advantage more prompt-aware.

There is still a tradeoff. A larger group gives a better estimate of the prompt-specific reward distribution, but it consumes more rollout compute. If the group is too small, the baseline is noisy. If it is too large, fewer distinct prompts fit into the same compute budget. Practical training chooses a group size that balances these factors.

DeepSeek's use of GRPO should therefore be read as an engineering compromise as well as an algorithmic idea. It removes the value model, keeps the reference model, and spends rollout compute on groups of completions whose rewards can be compared directly.

## 7.11 From toy code to a real LLM trainer

The chapter code is intentionally minimal, but it maps to a real trainer in a direct way. The main difference is scale and infrastructure, not the conceptual flow. A real trainer still samples completions, stores old log probabilities, verifies or scores completions, computes advantages, evaluates the current and reference policies, and applies a clipped policy-gradient update.

The first missing piece is generation. In the toy file, `sample_group` is a placeholder. In a real implementation, this function tokenizes prompts, calls the policy model's generate function with sampling enabled, and records the generated tokens. It must also record the old-policy log probability of each generated token because the PPO/GRPO ratio compares the new policy with the policy that produced the rollout.

The second missing piece is log-probability scoring. `sequence_logprobs` is also a placeholder. In a real implementation, it runs the model on the prompt-plus-completion sequences and gathers the log probability assigned to each generated token. This must be done carefully: the log probability for token n comes from the model output at the previous position, and prompt tokens should be masked out of the loss.

The third missing piece is batching. Real responses have different lengths. Some completions terminate early; others use the maximum token budget. A trainer pads sequences to a common length and uses masks so that padding does not affect the loss. Without correct masks, the model can receive gradients for tokens that were never generated.

The fourth missing piece is distributed execution. Large models cannot usually be trained with a single process that both generates rollouts and updates weights. Production systems separate rollout generation, reward computation, log-probability scoring, and optimization across devices. This chapter does not implement that machinery because it would obscure the algorithm, but the algorithmic data flow is the same.

| Toy function | Real trainer equivalent |
|---|---|
| sample_group | Tokenizer + policy.generate + storage of sampled tokens and old log probabilities |
| verify_math_answer | Task-specific verifier, unit tests, compiler, symbolic checker, or reward model |
| group_advantages | Prompt-group reward normalization with numerical safeguards |
| sequence_logprobs | Forward pass that gathers token log probabilities under policy/reference models |
| grpo_loss | Masked clipped surrogate objective with KL control and optimizer integration |

Thinking in this mapping helps prevent a common misunderstanding. The chapter code is not a toy algorithm. It is the small version of the algorithm. What it omits are production systems concerns: throughput, memory, robustness, safety filters, and monitoring.

### 7.11.1 Response format and verifier reliability

For RLVR, the response format is part of the training problem. If the verifier cannot reliably identify the final answer, the reward becomes noisy. A model might solve the problem correctly but phrase the answer in a way the verifier fails to parse. Or it might include several candidate numbers, causing the verifier to extract the wrong one.

One common solution is to require a structured final-answer format. For example, the prompt or system instruction may ask the model to place the final answer after a marker such as 'Final answer:'. The verifier can then parse only that region. The DeepSeek-R1 report discusses format rewards that encourage the model to separate reasoning from final answer regions. That is not cosmetic; it makes reward assignment more reliable.

The same principle applies to code. A code verifier may run unit tests, but it must know which file or function to test. If the model outputs prose, markdown fences, and code mixed, the evaluator must extract the runnable program. A brittle extractor can create a misleading reward signal. For this reason, RLVR datasets often define strict output protocols.

### 7.11.2 Monitoring training behavior

A GRPO/RLVR training run should be monitored with more than average reward. Average reward can rise while other behaviors degrade. For example, response length might explode, language quality might drop, or the policy might drift too far from the reference model.

Useful monitoring metrics include mean reward, pass rate, response length, KL divergence from the reference policy, invalid-format rate, duplicate-output rate, and reward variance within groups. For code tasks, hidden-test performance is important because visible-test reward can be exploited. For math tasks, a hold-out set with robust answer checking is necessary.

The DeepSeek-R1-Zero training curves are interesting because they show both improved accuracy and increased response length. The length increase is meaningful only because it accompanies better correctness. A model that simply writes more text without solving more problems is not improving reasoning. The reward and evaluation metrics must distinguish productive reasoning from verbosity.

### 7.11.3 What changes from PPO to GRPO in code?

If you already had a PPO trainer, moving to GRPO would not remove every PPO idea. The old-policy log probabilities, probability ratio, clipped objective, and reference KL remain. The main change is the source of the advantage.

In PPO, the advantage usually comes from returns minus value estimates. This requires a value model, value loss, value-model optimizer path, and sometimes generalized advantage estimation. In GRPO, the advantage comes from rewards within a sampled group. The trainer can remove the value model and replace the value-baseline computation with group normalization.

That replacement simplifies the system, but it also changes the data requirement. GRPO needs multiple completions per prompt so that the group baseline is meaningful. A PPO-style trainer that samples one response per prompt cannot compute a useful group-relative advantage unless it changes its rollout structure.

In summary: GRPO simplifies the model stack, not the entire training problem. It still needs careful rollout generation, reward design, KL control, and optimizer engineering.

## 7.13 A practical implementation checklist

If you were to extend the small chapter script into a more realistic experiment, the following checklist is a useful way to think about the system. Each item corresponds to a concrete engineering requirement, not just a conceptual idea.

* Start with the task distribution. GRPO with verifiable rewards needs prompts whose outcomes can be checked. For a math experiment, the dataset should include prompts and gold answers. For a code experiment, the dataset should include tests or another executable correctness signal. If the task cannot be verified, you need a learned reward model or a different alignment method.
* Define the response protocol before training. The model should know where to put its final answer. A verifier that searches arbitrary free-form text is fragile. A protocol such as 'write reasoning, then put the final answer after Final answer:' makes reward extraction more reliable. The protocol should be part of the prompt template and, if necessary, part of the reward.
* Create the reference policy. In most setups, this is a frozen copy of the model before RL begins. It should not be updated during the RL stage. Its job is to provide the KL anchor that keeps the trainable policy from drifting too far.
* Generate groups, not isolated responses. For each prompt, sample multiple completions from the current policy. Store the generated token IDs, attention masks, response masks, and old-policy log probabilities. These log probabilities are needed for the clipped ratio in the GRPO objective.
* Score the completions. Run the verifier or reward model on each completion. For RLVR tasks, record both the reward and useful diagnostics such as parse failure, invalid format, timeout, or unit-test failure. Diagnostics make it much easier to debug a reward that is not improving.
* Compute group-relative advantages. Normalize rewards within each prompt group. Watch for groups where all rewards are identical. If every completion gets the same reward, there may be little learning signal for that prompt. In practice, a curriculum or harder/easier prompt mix may be needed to keep reward variance useful.
* Compute current and reference log probabilities. Run the trainable policy and frozen reference model on the same generated sequences. Gather log probabilities only for generated response tokens. Apply masks so prompt tokens and padding do not contribute to the loss.
* Apply the GRPO loss. Use the probability ratio, clipping, advantages, and KL penalty. Track the mean reward, mean KL, response length, invalid-format rate, and loss. A rising reward with exploding KL is a warning sign. A stable KL with a flat reward may mean the policy is over-constrained or the reward is too sparse.
* Evaluate on hold-out tasks. The training reward is not enough. For math, use hold-out prompts and robust answer checking. For code, use hidden tests when possible. For general assistant behavior, use preference evaluation or human review. A model can overfit to the verifier or prompt distribution just as a supervised model can overfit to a dataset.

| Step | Question to ask before scaling |
|---|---|
| Task | Can the outcome be verified reliably? |
| Format | Can the final answer be parsed without ambiguity? |
| Rollout | Are old log probabilities and response masks stored correctly? |
| Reward | Are parse failures and invalid outputs tracked separately? |
| Advantage | Does each prompt group have useful reward variance? |
| KL | Is the policy staying close enough to the frozen reference? |
| Evaluation | Does held-out performance improve, not just training reward? |

This checklist also explains why production RL training is more complex than the short chapter code. The algorithm fits in a few functions; the reliable system around it requires careful data design, logging, distributed execution, evaluation, and safeguards.

## 7.14 Reading DeepSeek-R1 claims carefully

Because DeepSeek-R1 received a great deal of attention, it is worth ending the technical discussion by separating strong claims from overclaims. A strong claim is that GRPO plus verifiable rewards can induce reasoning behavior in a capable base model. DeepSeek-R1-Zero provides evidence for this claim. The model improved on reasoning benchmarks and showed longer, more self-corrective responses during RL training.

Another strong claim is that removing the value model can simplify large-scale RL training. GRPO replaces the learned value baseline with group-relative reward normalization. This reduces the model stack and avoids training a critic, while retaining clipping and reference-policy control.

A third strong claim is that distillation can transfer reasoning behavior into smaller models effectively. The R1-Distill results show that supervised fine-tuning on high-quality teacher-generated reasoning data can produce strong, smaller models.

The overclaim would be that verifiable rewards solve all alignment. They do not. They solve reward specification for tasks with objective checkers. The moment we move to open-ended helpfulness, creative writing, subtle factuality judgments, or safety preferences, deterministic verification becomes insufficient.

Another overclaim would be that chain-of-thought supervision is unnecessary in every setting. R1-Zero shows that supervised chain-of-thought is not required for the pure RL experiment to produce emergent reasoning behavior. But DeepSeek-R1 still uses cold-start supervised data and later supervised fine-tuning stages to improve readability and usability.

The balanced conclusion is more useful: RLVR gives us a powerful source of scalable reward for verifiable tasks; GRPO makes the RL update cheaper by removing the value model; staged supervised learning and preference rewards are still needed to turn the resulting behavior into a usable assistant.

## 7.15 Common implementation pitfalls

The minimal code is intentionally compact, but several details become important as soon as the implementation is scaled up.

1. Response masking is essential. The model should be updated on generated response tokens, not on the prompt tokens that were supplied by the dataset. Padding tokens should also be excluded from the loss.
2. Reward parsing must be robust. A model may include multiple numbers in a response. If the verifier extracts the wrong number, the reward becomes noisy. Many RLVR pipelines require the model to put the final answer in a specific format so the verifier can parse it reliably.
3. KL control must be monitored. If beta is too small, the policy can drift and exploit the reward. If beta is too large, learning may stall. Practical systems often adjust KL pressure based on observed divergence.
4. Group size is a tradeoff. Larger groups provide a better prompt-specific baseline, but they require more sampled completions per prompt. The compute budget must be allocated between the number of prompts, the number of completions per prompt, the response length, and the model size.
5. Reward hacking is always a risk. If the checker is incomplete, the model may learn to exploit it. For code, passing visible tests may not imply correctness on hidden tests. For math, answer extraction can be fooled by ambiguous output. Verifier design is part of model training, not a separate afterthought.

In the next chapter, we will use the outputs of a strong reasoning model as training data for smaller student models. That distillation step explains how DeepSeek-style reasoning behavior can be transferred into models that are cheaper to run and easier to deploy.

## 7.16 Summary

* LLM reinforcement learning treats the growing context as the state and each sampled token as an action.
* Policy-gradient methods update the probability of sampled actions in proportion to reward or advantage.
* PPO stabilizes policy updates with clipping and commonly uses a learned value model as a baseline.
* The value function Vϕ(s) is produced by a learned value model or value head trained from rollout returns.
* GRPO removes the learned value model by normalizing rewards within groups of completions sampled for the same prompt.
* GRPO is cheaper than PPO for the same total number of sampled completions because it eliminates the value-model path, not because it samples fewer responses.
* RLVR replaces learned reward models only when a task has objectively verifiable answers, such as many math and code tasks.
* DeepSeek-R1-Zero demonstrates emergent reasoning from GRPO plus verifiable rewards; DeepSeek-R1 adds staged SFT and RL to make the behavior more usable.
* The flow from R1-Zero into R1 is a data flow: filtered outputs become training data, not direct parameter transfer.
* The chapter code provides concrete hooks for verifier rewards, group-relative advantages, clipped GRPO loss, and a training-step skeleton.
