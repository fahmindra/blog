## 6.1 The data foundation: Preparing the TinyStories dataset

Before we can assemble our model, we must first prepare the fuel that will power its learning: the training data. This is a crucial, multi-step process that involves selecting an appropriate dataset, converting its raw text into a numerical format through tokenization, and saving it in a way that allows for efficient loading during training. Let’s begin by building a complete data preparation pipeline from scratch.

### 6.1.1 Choosing the right tools: The TinyStories dataset and TikToken

For our project, we need a dataset that is high-quality and large enough to facilitate meaningful learning, but small enough to be managed and trained on a single consumer GPU. This is where the TinyStories dataset comes in. Introduced by Microsoft researchers in the paper "TinyStories: How Small Can Language Models Be and Still Speak Coherent English?" (https://arxiv.org/abs/2305.07759), it is a collection of short stories generated by GPT-3.5 and GPT-4, hosted on Hugging Face at roneneldan/TinyStories. Its simple, coherent English allows even a small model to learn fundamental patterns of language, making it ideal for our purposes.

The next step is tokenization. While we could use the official DeepSeek tokenizer, its large vocabulary (100+k tokens) is optimized for a massive, multilingual corpus. For our smaller, English-only TinyStories dataset, a simpler approach is much more efficient. Therefore, we will use the standard 'gpt2' tokenizer from OpenAI's tiktoken library. This robust Byte-Pair Encoding (BPE) tokenizer has a vocabulary size of 50,257 tokens, saving us the effort of training our own while providing a solid foundation for the model to learn from. Our first step in the code will be to download the dataset and process it with this tokenizer.

### 6.1.3 The prepare.py script: A step-by-step walkthrough

Our first task is to create a script that automates the entire data preparation process. This script, which we'll call `prepare.py`, will handle four key responsibilities: downloading the raw text from Hugging Face, splitting it into training and validation sets, tokenizing the text into integer IDs, and saving the result in an efficient binary format. This script is the first of four core files we will create. It will be followed by `model.py`, where we will define our complete Mini-DeepSeek architecture; `train.py`, which will orchestrate the entire training process; and finally `sample.py`, a script to load our trained model and generate new text.

We will build `prepare.py` piece by piece to understand the logic behind each step. The full, executable code can be found in the chapter's GitHub repository. Let's begin with the initial imports and configuration.

**Listing 6.2 Imports and configuration in prepare.py**

The first part of our script defines the core components we'll use: the `DATASET_NAME` for the raw text and the `TOKENIZER_NAME` for converting that text into numbers. We also set a `VAL_RATIO`, reserving a small portion of the data for a validation set. This allows us to get an unbiased measure of how well our model is generalizing to new, unseen data during training.

With the configuration set, the main part of the script executes the data preparation workflow. It begins by loading the dataset, creating the train/validation splits, and then initializing the tokenizer. The `tiktoken` library makes it simple to load a standard, pre-trained tokenizer like gpt2. **Listing 6.3 Loading data and initializing the tokenizer**

The next step is to process the raw text of each story through the tokenizer. The `encode_corpus` function iterates through each text sample, encodes it into a list of integer IDs, and appends a special `End-of-Text (<|endoftext|>)` token. This `eot_token` acts as a separator, concatenating all the individual stories into a single, massive stream of tokens while still signaling document boundaries to the model during training.

The resulting stream of token IDs is stored as a numpy array of uint16 integers. Using uint16 (unsigned 16-bit integer) is a memory optimization; since our vocabulary size (50,257) is less than the maximum value for a uint16 (65,535), we can use this more compact data type instead of a standard 32-bit or 64-bit integer, saving significant disk space.

**Listing 6.4 The token encoding function**
```python
def encode_corpus(texts: List[str], enc: tiktoken.Encoding) -> np.ndarray:
    """Encodes a list of texts into a single flat stream of token IDs."""
    all_ids = []
    eot_token = enc.eot_token
    for text in tqdm(texts, desc="Encoding texts"):
        ids = enc.encode(text)
        all_ids.extend(ids)
        all_ids.append(eot_token)
    return np.array(all_ids, dtype=np.uint16)

# Inside main():
# 4. Tokenize the datasets
train_ids = encode_corpus([ex['text'] for ex in train_dataset], enc)
val_ids = encode_corpus([ex['text'] for ex in val_dataset], enc)
```

Finally, once we have our tokenized `train_ids` and `val_ids` arrays, we need to save them to disk. For large datasets, reading the entire file into memory during training is inefficient. A much better approach is to use memory-mapped files (memmap).

A memory-mapped file allows us to access a file on disk as if it were a part of the program's memory. The operating system handles the loading of data from disk into RAM in the background as it's needed. This is highly efficient for our training script, which only needs to load small batches of data at a time. The `write_memmap` function handles this process. We also save a `meta.json` file containing the `vocab_size` so our training script can easily access it.

**Listing 6.5 Saving data with memmap and metadata**
```python
def write_memmap(path: str, tokens: np.ndarray):
    """Writes a numpy array to a memory-mapped file."""
    # Shape is a tuple, so tokens.size, must be inside one
    arr = np.memmap(path, dtype=np.uint16, mode="w+", shape=(tokens.size,))
    arr[:] = tokens
    arr.flush()
    
# Inside main():
# 5. Write the tokenized data to binary files
write_memmap(os.path.join(OUTPUT_DIR, "train.bin"), train_ids)
write_memmap(os.path.join(OUTPUT_DIR, "val.bin"), val_ids)
 
# 6. Save metadata
meta = {"vocab_size": vocab_size}
with open(os.path.join(OUTPUT_DIR, "meta.json"), "w") as f:
    json.dump(meta, f)
print(f"\nPreparation complete. Data is saved in '{OUTPUT_DIR}'")
```

These files contain everything we need to begin training.

### 6.2.1 Building the components: RoPE, MLA, MoE, and MTP

With our configuration defined, we can build the core modules of our architecture. We will construct the model from the inside out, starting with the specialized helper classes for RoPE, Attention, and the MoE layer, before assembling them into the final TransformerBlock.

Our first component is the `RotaryPositionalEncoding` module. As we learned in Chapter 3, RoPE injects positional information by rotating pairs of dimensions in the Query and Key vectors. This helper class is responsible for pre-computing the necessary rotation frequencies (theta) and applying these rotations efficiently using complex number multiplication. A key update here is the addition of a `position_offset` argument, which is crucial for correctly applying positional encodings during cached inference when we are only processing one new token at a time.

**Listing 6.8 The RotaryPositionalEncoding Helper Module**
```python
class RotaryPositionalEncoding(nn.Module):
    def __init__(self, d_head: int, max_seq_len: int = 2048):
        super().__init__()
        self.d_head = d_head
        theta = 1.0 / (10000 ** (torch.arange(0, d_head, 2).float() / d_head))
        self.register_buffer('theta', theta)       
        positions = torch.arange(max_seq_len)
        freqs = torch.outer(positions, self.theta)
        freqs_cis = torch.polar(torch.ones_like(freqs), freqs)
        self.register_buffer('freqs_cis', freqs_cis, persistent=False)
 
    def forward(self, x: torch.Tensor, position_offset: int = 0):
        seq_len = x.shape[2]
        x_complex = torch.view_as_complex(x.float().reshape(*x.shape[:-1], -1, 2))        
        freqs_cis_slice = self.freqs_cis[position_offset : position_offset + seq_len]
        freqs_cis = freqs_cis_slice.unsqueeze(0).unsqueeze(0)
        x_rotated = x_complex * freqs_cis
        x_out = torch.view_as_real(x_rotated).flatten(3)
        return x_out.type_as(x)
```

Next, we implement the `DeepSeekAttention` module. This class is a direct translation of the complete Decoupled MLA+RoPE architecture we designed in chapter 3. It contains two parallel pathways: a Content Path that uses pure MHA for cache efficiency and a Position Path that applies RoPE to specialized projections.

The `__init__` method sets up all the necessary weight matrices for both paths. The forward method is where the decoupled MLA+RoPE logic comes together. It now accepts a `past_kv` argument, which will hold the cached tensors from previous steps during inference. It calculates the `position_offset` based on the length of this cache, ensuring that RoPE applies the correct rotations for new tokens. The content scores and position scores are calculated separately and then added together. Finally, it returns the output and the `new_cache` containing the updated latent tensors, ready for the next generation step.

**Listing 6.9 The DeepSeekAttention Module**
```python
class DeepSeekAttention(nn.Module):
    def __init__(self, args: ModelArgs):
        super().__init__()
        # ... (attribute initializations) …
        self.d_rope = args.d_rope
        # Content Path
        self.W_q_content = nn.Linear(args.d_model, args.d_model, bias=False)
        self.W_dkv_content = nn.Linear(args.d_model, args.d_latent, bias=False)
        self.W_uk_content = nn.Linear(args.d_latent, args.d_model, bias=False)
        self.W_uv_content = nn.Linear(args.d_latent, args.d_model, bias=False)
        # Position Path
        self.W_k_pos = nn.Linear(args.d_model, args.d_rope * args.num_heads, bias=False)
        self.W_q_pos = nn.Linear(args.d_model, args.d_rope * args.num_heads, bias=False)
        self.rope = RotaryPositionalEncoding(args.d_rope, max_seq_len=args.max_seq_len)
 
        # Output Projection
        self.W_o = nn.Linear(args.d_model, args.d_model, bias=False)
        self.dropout = nn.Dropout(args.dropout)
```

Now we are going to build a forward method for this `DeepSeekAttention`, as it handles the full decoupled logic for both training and cached inference.

During inference (when `past_kv` is provided), it calculates the `position_offset` and correctly appends the newly computed latent vectors (`c_kv_new` and `k_r_new`) to the cached tensors from previous steps. The content scores and position scores are calculated separately and then added together. Finally, it returns the output and the `new_cache` containing the updated latent tensors, ready for the next generation step.

**Listing 6.10 The forward method of DeepSeekAttention**
```python
def forward(self, x: torch.Tensor, attn_mask: torch.Tensor, 
            past_kv: Optional[Tuple[torch.Tensor, torch.Tensor]] = None, 
            position_offset: int = 0):
    B, S, D = x.shape
    past_len = past_kv[0].shape[1] if past_kv is not None else 0

    # --- Path A: Content Path —
    q_c = self.W_q_content(x).view(B, S, self.num_heads, self.d_head).transpose(1, 2)
    c_kv_new = self.W_dkv_content(x)
    c_kv = torch.cat([past_kv[0], c_kv_new], dim=1) if past_kv else c_kv_new
    k_c = self.W_uk_content(c_kv).view(B, past_len + S, ...).transpose(1, 2)
    v_c = self.W_uv_content(c_kv).view(B, past_len + S, ...).transpose(1, 2)        
    # --- Path B: Position Path —
    k_r_unrotated = self.W_k_pos(x).view(B, S, self.num_heads, self.d_rope).transpose(1, 2)
    q_r_unrotated = self.W_q_pos(x).view(B, S, self.num_heads, self.d_rope).transpose(1, 2)
    k_r_new = self.rope(k_r_unrotated, position_offset=position_offset)
    q_r = self.rope(q_r_unrotated, position_offset=position_offset)
    k_r = torch.cat([past_kv[1], k_r_new], dim=2) if past_kv else k_r_new
    # --- Combining Paths —
    content_scores = (q_c @ k_c.transpose(-2, -1)) / math.sqrt(self.d_head)
    position_scores = (q_r @ k_r.transpose(-2, -1)) / math.sqrt(self.d_rope)
    attn_scores = content_scores + position_scores
    # ... (softmax, dropout, output projection) ...        
    output = self.W_o(context_vector)
    new_cache = (c_kv, k_r)
    return output, new_cache
```

The next component is the `DeepSeekMoE` module. This class implements the full logic we developed in chapter 4, including fine-grained routed experts, shared generalist experts, and the auxiliary-loss-free load balancing mechanism. (Note: the MTP module below instantiates a TransformerBlock, which we will define in listing 6.15 once all the sub-components are in place.)

The `ExpertFFN` is a simple two-layer MLP that serves as the building block for both our routed and shared experts. The main `DeepSeekMoE` class then contains a list of these experts, separates them into `routed_experts` and `shared_experts`, and defines the gate (a linear layer) that will act as our router.

**Listing 6.11 The ExpertFFN and DeepSeekMoE __init__ method**
```python
class ExpertFFN(nn.Module):
    def __init__(self, d_model: int, hidden: int, dropout: float = 0.0):
        super().__init__()
        self.fc1 = nn.Linear(d_model, hidden, bias=False)
        self.fc2 = nn.Linear(hidden, d_model, bias=False)
        self.dropout = nn.Dropout(dropout)
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        return self.fc2(self.dropout(F.gelu(self.fc1(x))))
class DeepSeekMoE(nn.Module):
    def __init__(self, args: ModelArgs):
        super().__init__()
        self.n_routed = args.moe_n_routed_experts
        self.top_k = args.moe_top_k
        self.routed_experts = nn.ModuleList([
            ExpertFFN(args.d_model, args.moe_routed_hidden, args.dropout)
            for _ in range(self.n_routed)
        ])
        self.shared_experts = nn.ModuleList([
            ExpertFFN(args.d_model, args.moe_routed_hidden, args.dropout)
            for _ in range(args.moe_n_shared_experts)
        ])
        self.gate = nn.Linear(args.d_model, self.n_routed, bias=False)
        self.register_buffer("bias", torch.zeros(self.n_routed))
        self.bias_lr = 0.01
```

The forward method for our `DeepSeekMoE` module first handles the routing and dispatching of tokens. Every token is passed through the `shared_experts` to get a baseline output. In parallel, the gate calculates the router logits for the `routed_experts`. The current bias term is added to these logits to influence the top-k selection, and a softmax function converts the scores for the chosen experts into gate weights. Finally, a highly efficient sparse dispatch loop calculates the weighted output from the routed experts.

**Listing 6.12 The forward method of DeepSeekMoE (Routing and Dispatch)**
```python
def forward(self, x: torch.Tensor):
    B, S, D = x.shape
    x_flat = x.reshape(-1, D)        
    # Shared expert path (dense)
    shared_out = torch.zeros_like(x)
    for exp in self.shared_experts: shared_out += exp(x)        
    # --- Routed expert path (sparse) —
    router_logits = self.gate(x_flat)
    router_logits_with_bias = router_logits + self.bias.to(router_logits.dtype)
    top_k_logits, top_k_indices = torch.topk(router_logits_with_bias, self.top_k, dim=-1)
    gates = F.softmax(top_k_logits, dim=-1, dtype=torch.float).type_as(x)
    routed_out_flat = torch.zeros_like(x_flat)
    for i in range(self.n_routed):
        mask = (top_k_indices == i)
        row_idx, which_k = torch.where(mask)
        if row_idx.numel() == 0:
            continue
        w = gates[row_idx, which_k].unsqueeze(-1)
        exp_in = x_flat[row_idx]
        exp_out = self.routed_experts[i](exp_in)
        routed_out_flat.index_add_(0, row_idx, exp_out * w)        # ... (bias update and final output comes next)
```

After the main routing is complete, the forward method performs one final, crucial task during training: updating the dynamic bias for the next iteration. This entire block is wrapped in `if self.training:` so it is completely ignored during inference, and `torch.no_grad()` ensures that these operations do not affect the model's gradients. This separation is the key to the "loss-free" approach. The final output is the sum of the shared and routed expert outputs.

**Listing 6.13 The forward method of DeepSeekMoE (Bias Update and Final Output)**
```python
# ... (sparse dispatch logic from previous listing) ...
    if self.training:
        with torch.no_grad():
            avg_load = x_flat.size(0) * self.top_k / self.n_routed
            counts = torch.bincount(top_k_indices.flatten(), minlength=self.n_routed)
            violation = (avg_load - counts) / (avg_load + 1e-6)
            self.bias.add_(self.bias_lr * torch.tanh(violation))

    routed_out = routed_out_flat.view(B, S, D)
    return shared_out + routed_out
```

The final architectural component we need is the `MTPModule`. As we saw in chapter 5, this module is responsible for predicting one step further into the future in the causal chain. Our implementation is a simplified version for clarity. It contains a projection layer to merge the hidden state from the previous step with the embedding of the next token, and a dedicated `TransformerBlock` (described in listing 6.15) to process this merged information and produce a refined hidden state for the next MTP step.

**Listing 6.14 The Causal Multi-Token Prediction Module**
```python
class MTPModule(nn.Module):
    def __init__(self, args: ModelArgs):
        super().__init__()
        self.projection = nn.Linear(args.d_model * 2, args.d_model, bias=False)
        self.block = TransformerBlock(args)
    def forward(self, h_prev, next_token_embeds):
        x = torch.cat([h_prev, next_token_embeds], dim=-1)
        x = self.projection(x)
        B, S, D = x.shape
        # Create a causal mask for this step
        mask = torch.triu(torch.ones(S, S, device=x.device, dtype=torch.bool), diagonal=1)
        attn_mask = torch.zeros(S, S, device=x.device).masked_fill(mask, float('-inf'))        
        # We don't use KV cache inside MTP during training
        h_k, _ = self.block(x, attn_mask=attn_mask.unsqueeze(0).unsqueeze(0))
        return h_k
```

With all the specialized components defined, we can now assemble them into a standard `TransformerBlock`. This class follows the familiar "pre-norm" architecture, where a normalization layer is applied before the main operation (attention or feed-forward), and a residual connection is added after.

The forward method is crucial as it orchestrates the data flow and correctly threads the `past_kv` cache and `position_offset` through to the `DeepSeekAttention` module. This ensures that our KV cache works correctly during inference. The output of the attention block is added back to the input (first residual connection), and the same process is repeated for the `DeepSeekMoE` feed-forward layer.

**Listing 6.15 The TransformerBlock Module**
```python
class TransformerBlock(nn.Module):
    def __init__(self, args: ModelArgs):
        super().__init__()
        # Use LayerNorm for simplicity, though DeepSeek uses RMSNorm
        self.norm1 = nn.LayerNorm(args.d_model)
        self.attention = DeepSeekAttention(args)
        self.norm2 = nn.LayerNorm(args.d_model)
        self.feed_forward = DeepSeekMoE(args)
    def forward(self, x: torch.Tensor, attn_mask: torch.Tensor, 
                past_kv: Optional[Tuple[torch.Tensor, torch.Tensor]] = None, 
                position_offset: int = 0):
        # Attention block with residual connection
        h, new_cache = self.attention(self.norm1(x), attn_mask, past_kv, position_offset)
        x = x + h
        # Feed-forward block with residual connection
        x = x + self.feed_forward(self.norm2(x))
        return x, new_cache
```

### 6.2.2 The final architecture: MiniDeepSeek

We have now arrived at the top level of our architecture, the `MiniDeepSeek` class itself. This class acts as the main container, orchestrating all the components we have just built.

Its `__init__` method is responsible for creating the core parts of the model:

1. A shared token embedding layer (embed).
2. A stack of TransformerBlocks, which forms the main body or "trunk" of the model.
3. A final normalization layer (norm_f).
4. A shared output layer (lm_head) that projects the final hidden states to vocabulary logits.
5. A chain of MTPModules for multi-token prediction during training.

**Listing 6.16 Initializing the MiniDeepSeek Model**
```python
class MiniDeepSeek(nn.Module):
    def __init__(self, args: ModelArgs):
        super().__init__()
        self.args = args
        # Core model components
        self.embed = nn.Embedding(args.vocab_size, args.d_model)
        self.blocks = nn.ModuleList([TransformerBlock(args) for _ in range(args.n_layers)])
        self.norm_f = nn.LayerNorm(args.d_model)
        self.lm_head = nn.Linear(args.d_model, args.vocab_size, bias=False)
        # Multi-Token Prediction chain
        self.mtp_modules = nn.ModuleList([MTPModule(args) for _ in range(args.n_mtp_modules)])
    def causal_mask(self, S: int, device) -> torch.Tensor:
       mask = torch.triu(torch.ones(S, S, device=device, dtype=torch.bool), diagonal=1)
       return torch.zeros(S, S, device=device).masked_fill(mask, float('-inf'))
```

The forward method of our `MiniDeepSeek` model is where the distinction between our two primary modes of operation—training and inference—becomes explicit in the code. The method uses a simple but effective condition: `if targets is None`. During training, we will always provide the targets (the correct next tokens) to calculate the loss. During inference, we won't, which signals the model to use its optimized KV caching path.

#### The inference path

Let's first examine the inference path. This logic is designed for maximum efficiency during text generation. It correctly calculates the `position_offset` based on the length of the `past_kv_cache` and passes this information down to the RoPE module in each `TransformerBlock`. This ensures that even though we are only processing a single new token, its positional encoding is calculated correctly relative to its absolute position in the full sequence. After processing through all the blocks, it only computes the logits for the very last token, avoiding unnecessary computation.

**Listing 6.17 The Inference Path of the forward method**
```python
def forward(self, input_ids: torch.Tensor, targets: Optional[torch.Tensor] = None, 
            mtp_weight: float = 0.1, past_kv_cache: Optional[list] = None):
    B, S = input_ids.shape
    x = self.embed(input_ids)
    # --- Inference Path (with KV Cache) —
    if targets is None:
        position_offset = past_kv_cache[0][0].shape[1] if past_kv_cache else 0
        # Create a mask for the new token(s)
        mask = self.causal_mask(S + position_offset, x.device)[position_offset:, :]
        mask = mask.unsqueeze(0).unsqueeze(0)
        new_kv_cache = []
        for i, blk in enumerate(self.blocks):
            layer_past_kv = past_kv_cache[i] if past_kv_cache else None
            x, new_cache = blk(x, attn_mask=mask, past_kv=layer_past_kv, position_offset=position_offset)
            new_kv_cache.append(new_cache)
        x = self.norm_f(x)
        logits = self.lm_head(x[:, [-1], :])
        return logits, new_kv_cache
    # ... (Training Path comes next) …
```

#### The training path

When targets are provided, the model executes its training path. Here, the logic is different. We don't use the KV Cache, as we process the entire input sequence at once. The input `x` passes through the main stack of `TransformerBlocks` to produce the initial set of hidden states, `h_main`. From these, we compute the primary next-token prediction logits and the main loss (`loss_main`).

This is where the MTP chain comes into play. The model enters a loop, iterating through each `MTPModule`. In each step k, it uses the hidden state from the previous step (`h_prev`) and the ground-truth embeddings of the future tokens to predict the k+1-th token in the sequence. The loss for each MTP head is calculated and added to the `total_loss`, scaled by the `mtp_weight`. This combined loss provides the rich, multi-faceted gradient signal that helps the model learn to plan and forecast.

**Listing 6.18 The Training Path of the forward method**
```python
# ... (Inference Path from previous listing) ...

    mask = self.causal_mask(S, x.device).unsqueeze(0).unsqueeze(0)
    for blk in self.blocks:
        x, _ = blk(x, attn_mask=mask) # KV cache is ignored here
    h_main = self.norm_f(x)
    logits_main = self.lm_head(h_main)
    # Standard next-token prediction loss
    loss_main = F.cross_entropy(...)
    total_loss = loss_main
    # MTP causal chain
    h_prev = h_main
    for k, mtp_block in enumerate(self.mtp_modules, start=1):
        if S <= k + 1: break # Cannot predict beyond sequence
        # Slice inputs for the k-th prediction depth
        h_prev_shifted = h_prev[:, :-(k+1), :]
        next_tokens_embed = self.embed(input_ids[:, k:-1])
        h_k = mtp_block(h_prev_shifted, next_tokens_embed)
        logits_k = self.lm_head(self.norm_f(h_k))
        # Calculate loss for this MTP head
        loss_k = F.cross_entropy(...)
        total_loss += mtp_weight * loss_k
    return {"logits": logits_main, "loss": total_loss}
```

### 6.3.2 Loading data and scheduling the learning rate

Before we start the main training loop, we need two key helper functions. The first is `get_batch`. This function is responsible for loading a mini-batch of data for a single training step. It opens the memory-mapped `.bin` file for the specified split (either 'train' or 'val'), randomly selects `batch_size` starting positions, and slices out the corresponding input sequences (`x`) and target sequences (`y`). Using `pin_memory()` is a performance optimization that allows for faster data transfer from the CPU's RAM to the GPU's VRAM.

**Listing 6.20 The get_batch data loading function**
```python
def get_batch(split: str):
    data = np.memmap(os.path.join(data_dir, f'{split}.bin'), dtype=np.uint16, mode='r')
    ix = torch.randint(len(data) - block_size, (batch_size,))
    x = torch.stack([torch.from_numpy((data[i:i+block_size]).astype(np.int64)) for i in ix])
    y = torch.stack([torch.from_numpy((data[i+1:i+1+block_size]).astype(np.int64)) for i in ix])
    if 'cuda' in device:
        return x.pin_memory().to(device, non_blocking=True), y.pin_memory().to(device, non_blocking=True)
    else:
        return x, y
```

The second helper function is `get_lr`, which implements a learning rate scheduler. Instead of using a fixed learning rate throughout training, it's common practice to vary it. Our scheduler implements a cosine decay with a warmup strategy, which is a highly effective and popular choice for training Transformers.

The process has three phases:

1. Warmup: For the first `warmup_iters` steps, the learning rate increases linearly from 0 to its peak value. This helps stabilize the model in the early stages of training when the weights are still random.
2. Cosine Decay: After the warmup, the learning rate gradually decreases, following a cosine curve. This allows the model to make large updates early on and then fine-tune its parameters with smaller updates as it gets closer to a good solution.
3. Final Cooldown: The learning rate eventually decays to a small minimum value (`min_lr`) for the remainder of training.

**Listing 6.21 The learning rate scheduler function**
```python
def get_lr(it):
    # 1) Linear warmup for warmup_iters steps
    warmup_iters = 200
 
    if it < warmup_iters:
        return learning_rate * it / warmup_iters
    # 2) If it > lr_decay_iters, return min_lr
    lr_decay_iters = max_iters
    min_lr = learning_rate / 10
    if it > lr_decay_iters:
        return min_lr
    # 3) In between, use cosine decay
    decay_ratio = (it - warmup_iters) / (lr_decay_iters - warmup_iters)
    coeff = 0.5 * (1.0 + math.cos(math.pi * decay_ratio))
    return min_lr + coeff * (learning_rate - min_lr)
```

> **A note on the `min_lr` branch:** With `lr_decay_iters = max_iters`, the training loop exits before it ever exceeds `lr_decay_iters`, so the `if it > lr_decay_iters: return min_lr` branch is a defensive guard rather than a path the code hits in normal operation. It ensures `get_lr` remains well-defined if you later extend `max_iters` past `lr_decay_iters` — for example, when resuming from a checkpoint to fine-tune for a few extra iterations at the cool-down rate. If you would prefer the cool-down to actually run during the main training, set `lr_decay_iters` to roughly `0.9 * max_iters`.

## 6.4 The engine of scale: Understanding DualPipe Parallelism

We have successfully built and configured a complete pipeline to train our MiniDeepSeek model on a single GPU. However, our `train.py` script is just one piece of a much larger puzzle. The real DeepSeek-V3 was trained on a massive cluster of over 2,000 GPUs, a feat made possible by a sophisticated training strategy called DualPipe Parallelism.

Before we dive into this advanced technique, let's take a moment to build a solid foundation by looking at the core ideas of parallel computing for deep learning. In this section, we'll walk through the essential strategies used to distribute the workload of training a large language model across multiple GPUs. This will lead us to the innovations that make systems like DualPipe possible.

As illustrated in figure 6.2, parallel computing in deep learning can be broadly divided into two main categories, each solving a different problem.

![Title: dimensions:[378x142] - Description: 062.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/06__image002.png)
*Figure 6.2 The two primary types of parallel computing in deep learning. Data Parallelism accelerates training by splitting the data, and Model Parallelism bypasses memory limits by splitting the model itself. In modern LLM training, these are not either/or choices, but complementary techniques combined to simultaneously overcome the dual constraints of time and hardware.*

At its core, parallel computing is the principle of using multiple processors (in our case, GPUs) simultaneously to solve a single, larger problem more quickly. Training a massive language model is an incredibly slow process. Processing trillions of tokens on a single GPU could take months or even years. The core question for scaling up is: "How can we use more hardware to make training faster?"

Before we dive into how to use multiple GPUs, we must first understand a crucial technique that helps us maximize the potential of even a single GPU: **Gradient Accumulation**. This concept is a prerequisite for understanding both Data and Model Parallelism, as it introduces the core problem of memory limitations and provides a software-based solution.

### 6.4.1 The standard training loop and its memory limit

To understand why we need techniques like gradient accumulation, let's first revisit the standard procedure for a single weight update in a typical training loop. Our primary goal is to minimize the model's error (loss) by adjusting its internal parameters (weights and biases).

The process for a single step is as follows:

1. **Get a Mini-Batch:** We take a small group of training examples (e.g., 32 images or text sequences) and their corresponding labels. We use a mini-batch instead of a single example because the gradient from just one sample can be very noisy and misleading.
2. **Forward Pass:** We feed the entire mini-batch through the model to get a set of predictions.
3. **Calculate Loss:** We compare the model's predictions to the true labels. The loss function averages the "wrongness" across all examples in the mini-batch.
4. **Backward Pass:** This is the core of learning. Calling `loss.backward()` calculates the gradients of the loss with respect to every single weight in the model. This gradient represents the average best direction to move the weights to reduce the loss for that specific mini-batch.
5. **Update Weights:** The optimizer (`optimizer.step()`) takes these calculated gradients and nudges all the model's weights slightly in the correct direction.
6. **Zero Gradients:** Finally, we clear out the old gradients (`optimizer.zero_grad()`) so they don't interfere with the calculations for the next mini-batch.

This loop forward, backward, step, zero is repeated for thousands of mini-batches. This process works perfectly, but it has a hard physical limitation: the entire mini-batch must fit into your GPU's memory (VRAM).

Modern models are huge, and their inputs can also consume a lot of memory. A single high-resolution image or a long text sequence requires significant VRAM. Furthermore, to get a stable and accurate gradient, it is often recommended to use very large batch sizes, 256, 512, or even thousands of examples. If you have a GPU with limited memory and you try to use a batch size that is too large, you will be met with the CUDA "out of memory" error.

This leads us to the core problem that gradient accumulation solves: **How can we get the benefit of a large batch size (a more stable, averaged gradient) without needing the massive VRAM to process it all at once?**

The solution lies in a simple but powerful insight. The final weight update is mathematically equivalent whether you calculate the gradients for a large batch all at once, or if you calculate the gradients for smaller "chunks" of that batch and simply add them up before making the single update step. This is the principle of **Gradient Accumulation**.

To understand this mathematically, let's consider the gradient. The gradient for a single sample `x`, which we'll call `g`, is the vector of partial derivatives of the loss function `L` with respect to all the model's parameters `θ` (theta). It tells us how the loss changes as we change each weight by a tiny bit.

```math
g = ∇_θ L(x)
```

For a standard mini-batch of `B` samples, the final gradient used for the weight update is simply the average of the gradients for each sample in the batch:

```math
g_batch = (1/B) * (g_1 + g_2 + ... + g_B)
```

Now, imagine our desired batch size `B` (e.g., 64) is too big for our GPU, which can only handle a smaller "physical" batch size `b` (e.g., 16). We can split our large "effective" batch into N smaller chunks, where N = B / b. In our example, 64 / 16 = 4 chunks.

Gradient accumulation states that we can get the same g_batch by first calculating the average gradient for each smaller chunk (g_chunk1, g_chunk2, etc.), summing them up, and then averaging that final sum. The key is that PyTorch's `loss.backward()` call naturally accumulates (adds) new gradients to the `.grad` attribute of each parameter if you don't zero them out.

### 6.4.2 Gradient accumulation in code

Let's compare the standard training loop with a gradient accumulation loop to see this principle in action. Our goal is to achieve an Effective Batch Size of 64 using a Physical Batch Size of 16, which requires 4 Accumulation Steps.

A standard training loop would try to process all 64 samples at once. If this fits in memory, the update happens on every single iteration.

**Listing 6.25 Standard Training Loop (Simplified)**
```python
# Desired batch_size = 64, but this would crash on a small GPU
for batch in dataloader: # Dataloader provides batches of 64
    # 1. Forward pass
    outputs = model(batch)
    loss = loss_function(outputs, batch.labels)
    # 2. Backward pass (calculates gradients for the full batch)
    loss.backward()
    # 3. Update weights (takes a step)
    optimizer.step()
    # 4. Zero the gradients for the next batch
    optimizer.zero_grad()
```

Now, let's look at the gradient accumulation version. The key difference is the conditional execution of `optimizer.step()` and `optimizer.zero_grad()`.

**Listing 6.26 Gradient Accumulation Loop (Simplified)**
```python
accumulation_steps = 4
# Dataloader now provides small batches of 16
for i, batch in enumerate(dataloader):
    # 1. Forward pass (on the small batch)
    outputs = model(batch)
    loss = loss_function(outputs, batch.labels)
    # Normalize the loss
    loss = loss / accumulation_steps
    # 2. Backward pass (accumulates gradients for the small batch)
    loss.backward()
    # 3. Update weights only after enough steps
    if (i + 1) % accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad()
```

#### Let's trace what happens

The magic of gradient accumulation happens because the gradients are not cleared after every backward pass. Let's trace the first four iterations:

* Iteration 1 (i=0): `loss.backward()` calculates gradients for the first batch of 16. These are now stored in each parameter's `.grad` attribute. The `if` condition is false, so we skip the `step()` and `zero_grad()`. The gradients are not erased.
* Iteration 2 (i=1): `loss.backward()` is called again on the second batch of 16. PyTorch automatically adds (accumulates) these new gradients to the ones already stored. The `.grad` attributes now hold the sum of gradients from 32 examples. The `if` condition is still false.
* Iteration 3 (i=2): The process repeats. The `.grad` attributes now hold the sum of gradients from 48 examples. The `if` condition is still false.
* Iteration 4 (i=3): `loss.backward()` is called for the fourth time. The `.grad` attributes now hold the sum for all 64 examples. The `if` condition (3 + 1) % 4 == 0 is now TRUE!
    * `optimizer.step()` is called. The optimizer looks at the accumulated gradients (which represent the entire effective batch of 64) and updates the model's weights.
    * `optimizer.zero_grad()` is finally called, clearing the `.grad` attributes so we can start accumulating for the next effective batch.

The result? We successfully updated our model using a gradient calculated from 64 examples, even though we only ever had 16 examples in memory at any given time.

#### The pros and cons of gradient accumulation

The gradient accumulation has a clear advantage: it allows us to train with large, effective batch sizes that would otherwise be impossible on memory-constrained hardware. Since larger batch sizes often lead to more stable training and can result in a model that generalizes better, this is a significant benefit. The gradients calculated from a larger effective batch provide a more accurate estimate of the "true" gradient over the dataset, reducing noise in the weight updates and leading to a smoother path down the loss landscape.

However, this comes with one major, unavoidable drawback: it increases training time. In our example, we perform four forward passes and four backward passes for every single optimizer step. A GPU that could handle all 64 samples natively would only perform one forward and one backward pass for that same update. We are explicitly trading time for memory.

As we saw, gradient accumulation is a clever trick to work around memory limits, but it actually makes the training loop slower. This brings us to our next critical question. If we have more hardware, how can we use it to not just fit bigger batches, but to make the entire process fundamentally faster? This is where parallel computing comes in.

### 6.4.3 Data parallelism: The "more GPUs, less time" approach

Training a large model is slow, and gradient accumulation, while solving the memory problem, actually increases training time. The most direct way to accelerate training is to use more hardware. This leads us to our first major parallel computing strategy: Data Parallelism.

The core idea is intuitive: if one GPU can process a batch of 16 samples, then four GPUs can process four batches of 16 simultaneously. By splitting a large batch of data across multiple devices, we can perform the forward and backward passes for all chunks in parallel. The goal is to compute the gradient for the entire batch much faster than a single GPU could.

![Title: dimensions:[378x137] - Description: 063.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/06__image005.png)
*Figure 6.3 The workflow of Data Parallelism. A large data batch is sharded into mini-batches, and each GPU processes one mini-batch using its local copy of the model.*

As illustrated in figure 6.3, a large data batch is first divided into mini-batches, with each mini-batch assigned to a different GPU. Crucially, each GPU holds an identical copy of the entire model. Once the mini-batches are distributed, every GPU performs its own forward and backward pass on its local slice of data. After all devices finish computing their gradients, those gradients are gathered, aggregated, and averaged across GPUs. This collective synchronization ensures that all model replicas apply the same parameter update, keeping their weights perfectly aligned for the next iteration.

In PyTorch, there are two primary implementations of this strategy: the older `nn.DataParallel` (DP) and the modern, industry-standard `nn.DistributedDataParallel` (DDP). While they share the same high-level goal, their mechanics are different. Let's examine how each one works.

### 6.4.4 Method 1: nn.DataParallel (DP)

DataParallel, often referred to as DP, is the older, simpler-to-implement method. It operates using a centralized model where one GPU is designated as the main or "default" device (typically cuda:0).

The process for a single training step using DP is as follows:

1. **Scatter**: The full data batch is first loaded onto the main GPU. The main GPU then scatters smaller chunks of the batch to all other available GPUs. It also replicates the entire model from itself to every other GPU on every single forward pass.
2. **Parallel Forward Pass**: Each GPU performs a forward pass on its slice of the data.
3. **Gather**: The outputs (logits) from each GPU are all sent back to the main GPU.
4. **Centralized Computation**: The main GPU gathers all the outputs, calculates the total loss, and performs the backward pass to compute the gradients for the entire batch. All of this work happens only on the main GPU.
5. **Update and Repeat**: The main GPU uses the final gradients to update its copy of the model's weights. For the next batch, this newly updated model must be copied out to all the other GPUs again.

While simple to implement, this centralized design has significant drawbacks that make it unsuitable for serious LLM training. The main GPU becomes a massive bottleneck, as it does far more work than any other device, and all network traffic is funneled through this single point. This leads to severe load imbalance and poor scalability. As a result, DP is rarely used for modern large-scale training.

### 6.4.5 Method 2: nn.DistributedDataParallel (DDP)

DistributedDataParallel, or DDP, is the modern, industry-standard approach that solves the fundamental flaws of DP. Instead of a centralized model, DDP treats every GPU as an independent node, launching a separate process for each one. This decentralized, "committee" approach leads to a well-balanced and highly efficient system that scales far better than DP in practice.

Here’s the step-by-step DDP process:

1. **Setup**: Before training begins, a copy of the model is placed on each GPU in its own independent process. Each process also initializes its own optimizer, which is responsible only for the model replica on its GPU.
2. **Shard the Data**: The dataset is sharded, meaning each process loads its own unique slice of the data for the batch directly. There is no central GPU managing data distribution.
3. **Parallel Forward & Backward Pass**: Each process (on each GPU) feeds its data through its local model copy, calculates its local loss, and performs a backward pass to compute gradients for its local weights. The per GPU workload is evenly distributed because every process owns the same slice of the computation.
4. **The All-Reduce Step**: This is the core of DDP's efficiency. As soon as a process finishes calculating a gradient for a layer, it immediately starts communicating that gradient to its peers. They use a highly efficient communication algorithm (like Ring-AllReduce) to average the gradients across all processes. This communication is often overlapped with the backward pass computation, hiding much of its latency.
5. **Parallel Update**: By the time the backward pass is finished, every single process has the identical, averaged gradient for the entire global batch. They can all run the optimizer step locally and in parallel on their copy of the weights.
6. **Stay in Sync**: Because all model replicas started with the same weights and applied the same update, they remain synchronized, ready for the next batch. No central bottleneck, no "boss" GPU needed.

### 6.4.6 Practical implementation of DDP

DDP requires a slightly different way of launching our training script and a few setup steps within our code. DDP requires a "launcher" that sets up the distributed environment. This launcher is responsible for creating one process per GPU and assigning a unique ID (called a "rank") to each one. The modern, recommended way to do this is with torchrun.

```bash
# uses torchrun to use all 4 available GPUs on this machine
# and to run the 'train.py' script in each of the 4 processes.
torchrun --nproc_per_node=4 train.py --args
```

Inside the training script, we need to add a few lines of code to enable DDP:

To use Distributed Data Parallelism effectively, we begin by initializing the process group. This step establishes the communication channel among all participating processes, the distributed equivalent of setting up a conference call.

Next, each process must be assigned to a specific GPU, and tools like torchrun provide the `LOCAL_RANK` environment variable to ensure that every process is correctly pinned to its designated device. Once the devices are mapped, we wrap our standard PyTorch model inside the DistributedDataParallel (DDP) class. This wrapper is critical, as it intercepts the backward pass and automatically manages the underlying All-Reduce operations required for gradient synchronization. Finally, to ensure that every GPU receives a unique portion of the dataset, we rely on a DistributedSampler, which partitions the data so that each process works on a distinct shard during every training step.

**Listing 6.27 Key DDP setup steps in PyTorch**
```python
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data.distributed import DistributedSampler
# 1. Initialize the process group
dist.init_process_group(backend='nccl')
# 2. Assign the device for the current process
local_rank = int(os.environ['LOCAL_RANK'])
torch.cuda.set_device(local_rank)
# 3. Wrap the model
model = MiniDeepSeekM().to(local_rank)
ddp_model = DDP(model, device_ids=[local_rank])
# 4. Use the distributed sampler for the dataloader
train_sampler = DistributedSampler(train_dataset)
train_dataloader = DataLoader(train_dataset, batch_size=..., sampler=train_sampler)
```

After this setup, the rest of your training loop looks almost identical to a single-GPU loop, but we use `ddp_model` instead of `model`. The DDP wrapper handles all the complex communication behind the scenes.

### 6.4.7 DDP + gradient accumulation

By this point, we can combine the solutions to our two major bottlenecks: the memory limitation and the time limitation. Distributed Data Parallelism addresses the time bottleneck by spreading computation across multiple GPUs, while Gradient Accumulation addresses the memory bottleneck by allowing us to simulate a very large batch size that would never fit into a single device. Together, these two techniques form the foundational scaling strategy behind nearly all large-scale LLM training.

Consider a practical scenario. Suppose our target effective batch size is 4096, which is often required for stable optimization, yet a single A100 GPU can only hold a physical batch of 8 samples. At the same time, training on one GPU is too slow, but we have access to a machine with eight A100 GPUs. By running our training script under DDP on all eight devices, we immediately scale our throughput by roughly a factor of eight, since each GPU processes its own batch independently. Because each GPU handles a batch of 8 samples, the total amount of data processed in a full forward and backward pass for the global batch size is:

```math
global_batch_size = per_gpu_batch_size * num_gpus 
global_batch_size = 8 * 8 = 64 
```

However, our target remains an effective batch size of 4096. To reach it, we rely on Gradient Accumulation. The number of accumulation steps required is simply:

```math
accumulation_steps = target_batch_size / global_batch_size 
accumulation_steps = 4096 / 64 = 64 
```

This means that every GPU performs its forward and backward pass 64 times, accumulating gradients locally, before performing a single synchronized update across devices. The result is that we achieve the stability benefits of a batch size of 4096 while training at the throughput of eight GPUs operating in parallel. Together, these two techniques address both the memory and the time bottlenecks.

## 6.5 Model parallelism

We have successfully combined Data Parallelism (DDP) and Gradient Accumulation to solve the bottlenecks of training time and batch size memory. This is the foundation of most large-scale training. However, it relies on one critical assumption: that a complete copy of the model can fit onto a single GPU.

As models grow into the hundreds of billions of parameters, this assumption breaks down. The sheer size of the model's parameters (weights), its gradients, and the optimizer states (e.g., Adam's moments) can easily exceed the VRAM of a single, even top-of-the-line, GPU. A 671B parameter model like DeepSeek-V3 requires over 350GB of VRAM just for the weights (at half-precision), far more than any single GPU can provide.

Data Parallelism cannot solve this. In fact, since DDP replicates the entire model on every GPU, it offers no memory savings for the model itself. When the model is too big to fit, we must turn to a new family of techniques collectively known as Model Parallelism. The core idea is simple: instead of splitting the data, we split the model itself across multiple GPUs.

### 6.5.1 Key terminology for model parallelism

Before we explore the different strategies for splitting a model, let's define a few key terms that are central to understanding how these techniques work and how they are evaluated.

* **Communication:** Any time GPUs need to exchange data (tensors) with each other. This is the primary source of overhead in parallel computing. Different strategies have different communication patterns (e.g., the All-Reduce we saw in DDP). The goal of any advanced parallelism technique is always to minimize the amount of communication or, even better, to overlap it with computation so that it becomes "free."
* **Micro-batch:** A small chunk of a data batch. Instead of processing a batch of 32 all at once, you might break it into 8 micro-batches of size 4. This concept is essential for making Pipeline Parallelism efficient.
* **Stage:** A part of a larger computation pipeline. In model parallelism, a stage is typically a group of model layers that are placed on a single GPU. Data (activations) flows from Stage 0 (GPU 0) to Stage 1 (GPU 1), and so on.
* **The "Bubble" (or Pipeline Bubble):** This is the single most important concept for understanding the efficiency of Pipeline Parallelism. It refers to the idle time on GPUs when they are waiting for data from a previous stage to arrive (during the forward pass) or for a gradient signal from a subsequent stage to arrive (during the backward pass). A large bubble means your expensive GPUs are sitting around doing nothing, which is incredibly inefficient. The primary goal of advanced pipeline techniques is to minimize this bubble.

### 6.5.2 Pipeline parallelism: The assembly line approach

The most intuitive and widely used form of model parallelism is Pipeline Parallelism. Instead of splitting the computations within a layer (like Tensor Parallelism), Pipeline Parallelism splits the model between layers. It partitions the sequence of Transformer blocks into several contiguous chunks called stages, and places each stage on a different GPU. The data then flows from one GPU to the next, like a product moving down an assembly line.

### 6.5.3 The naive schedule (GPipe) and the pipeline bubble

The foundational paper that introduced this technique is Google's "GPipe" (Part Four: https://arxiv.org/pdf/1811.06965). Its implementation highlights the central challenge of pipeline parallelism: the "bubble." To make the pipeline work, a large training batch must be split into smaller micro-batches. The pipeline is then "filled" by feeding these micro-batches into the first stage one after another.

We have created a hands-on visualization of this process, training a simple model across four GPUs to generate the timeline charts you see here. You can find the full code for this experiment, which was run on four NVIDIA RTX 4090 GPUs, in the book's GitHub repository. We encourage you to explore it to get a feel for how these schedules are implemented.

Let's begin by analyzing the first phase of the GPipe schedule.

#### Phase 1: The forward pass (filling the pipe)

In this phase, all micro-batches are pushed through the pipeline from the first stage to the last. The process is sequential: GPU 1 cannot start working on a micro-batch until GPU 0 has finished with it and passed along the resulting activations.

As you can see in the "Forward Pass (Fill Phase)" diagram (figure 6.4), this creates a significant stagger. GPU 1 is idle for one time step, GPU 2 is idle for two, and GPU 3 is idle for three, all waiting for the first micro-batch to propagate through the system. This initial period of idle time at the start of the forward pass is the first half of the pipeline bubble. Only after the pipe is "full," meaning every GPU is working on a different micro-batch, does the system reach maximum utilization.

![Title: dimensions:[378x161] - Description: 064.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/06__image006.png)
*Figure 6.4 The forward pass in a naive pipeline schedule. Each GPU must wait for the previous one to complete its work on a micro-batch. The white area to the left of the bars represents the initial idle time (the "bubble") as the pipeline fills up.*

This period of idle time is a direct consequence of the sequential data dependency. Let's trace the timesteps of the first few micro-batches to make this concrete. We'll assume we have 8 micro-batches (labeled 1 through 8) and 4 GPUs:

**At Time Step 1:** GPU 1 processes micro-batch 1. GPUs 2, 3, and 4 are completely idle, waiting for data.

**At Time Step 2:** GPU 1 sends the output of micro-batch 1 to GPU 2. While GPU 2 works on micro-batch 1, GPU 1 can now start processing micro-batch 2. GPUs 3 and 4 are still idle.

**At Time Step 3:** The pipeline continues to fill. GPU 3 starts on micro-batch 1, GPU 2 starts on micro-batch 2, and GPU 1 starts on micro-batch 3. GPU 4 remains idle.

**At Time Step 4:** The pipeline is finally "full." GPU 4 begins work on micro-batch 1, and all other GPUs are processing subsequent micro-batches.

As the figure clearly shows, it takes N-1 time steps (where N is the number of GPUs) before the last GPU even begins its work. This startup latency is a source of inefficiency.

#### Phase 2: The backward pass (draining the pipe)

Once all the micro-batches have completed their forward pass, the process reverses for the backward pass. Gradients must be calculated starting from the last stage (GPU 4) and propagated backward through the pipeline. This is because the gradient calculation for any given layer depends on the gradients from the layer that follows it.

This creates a "draining" phase that mirrors the initial "fill" phase. As shown in Figure 6.5, the early-stage GPUs (like GPU 1 and 2) are forced to sit idle, waiting for the gradient information to flow back to them from the later stages. This idle time at the end of the backward pass constitutes the second half of the pipeline bubble.

![Title: dimensions:[378x161] - Description: 065.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/06__image007.png)
*Figure 6.5 The backward pass in a naive pipeline schedule. The process moves in reverse, starting from the last GPU. The white area to the right of the bars represents the idle time as the pipeline drains.*

This "draining" process is also sequential. The backward pass for micro-batch 8 must be completed on GPU 4 before its input gradient can be sent to GPU 3. Only then can GPU 3 compute its backward pass for micro-batch 8, and so on. This dependency means that GPU 1, for example, must wait for all other GPUs to finish their backward passes on a given micro-batch before it can begin its own work.

#### The problem: Visualizing the combined bubble

When we combine the forward and backward passes into a single schedule, the inefficiency of the GPipe approach is clear. The model executes the entire forward pass for all micro-batches first, and only then begins the entire backward pass.

![Title: dimensions:[378x161] - Description: 066.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/06__image008.png)
*Figure 6.6 The combined GPipe schedule, showing the pipeline bubble. The large block of idle time (white space) between the forward and backward passes represents a massive amount of wasted computational resources.*

As Figure 6.6 clearly illustrates, the GPUs are not working for a significant portion of the total time. The idle time from the "fill phase" at the start and the "drain phase" at the end combine to form a large "bubble" in the middle of the computation schedule. This bubble represents valuable GPU resources sitting completely unused. The size of this bubble grows proportionally with the number of pipeline stages (GPUs), making this naive schedule extremely inefficient for deep models that require many stages. This is the flaw that the next generation of pipeline parallelism techniques set out to solve.

### 6.5.4 The 1F1B schedule: Eliminating the bubble

The inefficiency of the GPipe schedule led to the development of a much smarter, interleaved scheduling approach, most commonly known as 1F1B (One Forward, One Backward). This is the critical optimization that makes modern pipeline parallelism practical and is used by all major training frameworks.

The core intuition behind 1F1B is simple: Why wait? Instead of running all forward passes and then all backward passes, why not start the backward pass for a micro-batch as soon as its forward pass is complete? This allows GPUs to interleave forward and backward computations, keeping them busy instead of idle.

This process breaks the pipeline execution into three distinct phases: a brief "warm-up" to fill the pipe, a highly efficient "steady state" where GPUs are constantly working, and a brief "cool-down" to drain the final gradients.

![Title: dimensions:[378x161] - Description: 067.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/06__image009.png)
*Figure 6.7 The 1F1B interleaved schedule. Forward (blue) and backward (green) passes alternate during the steady state. Note: any small gaps visible in this rendered timeline are artifacts of the profiling resolution, not the schedule itself. In a true 1F1B, each GPU executes its remaining backward passes back-to-back during cool-down.*

As you can see in figure 6.7, the visual difference is dramatically denser than GPipe. The large blocks of idle time are gone, replaced by a tightly packed sequence of alternating forward and backward tasks. Let's trace how this works to understand its benefits.

The 1F1B schedule is not a single phase but a sequence of three. Let's analyze the timeline for a GPU in the middle of the pipeline, like GPU 2, to see how it operates.

1. **Phase 1: The Warm-up.** The pipeline still needs to be filled initially. GPU 2 must wait for GPU 1 to process the first micro-batch. It performs a few forward passes (e.g., for micro-batches 1 and 2) to push data down the pipeline. This initial, unavoidable latency is the "warm-up" bubble, seen as the small idle area at the start of each GPU's timeline.
2. **Phase 2: The Steady State.** This is where the core optimization happens. Once a GPU (e.g., GPU 2) completes a forward pass for micro-batch i, it immediately looks for other work to do. Instead of waiting to start the forward pass for micro-batch i+1 (which might not have arrived yet), it checks if the gradient for an earlier micro-batch (say, i-1) has arrived from the next stage (GPU 3). If it has, the GPU immediately begins the backward pass for that micro-batch. It is now constantly alternating: Forward -> Backward -> Forward -> Backward. This interleaving keeps the GPU utilized for the majority of the training step.
3. **Phase 3: The Cool-down.** Once all forward passes are complete, the GPU must finish the remaining backward passes that are still propagating up the pipeline. This creates a small "cool-down" bubble at the very end, as seen in the figure.

By eliminating the large central bubble, the 1F1B schedule ensures that GPUs are kept busy for a much larger portion of the total step time. This directly translates to huge improvements in real-world training speed and throughput. The more micro-batches there are, the longer the "steady state," and the smaller the relative impact of the warm-up and cool-down bubbles on overall efficiency.

However, it's crucial to understand a theoretical nuance here. The maximum achievable GPU utilization, or efficiency, of a pipeline is determined by the ratio of computation time to the total time (computation + bubbles). The formula for pipeline efficiency is often expressed as:

![Title: dimensions:[378x22] - Description: 068.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/06__image010.png)

Theoretically, for a given number of micro-batches and stages, the maximum achievable utilization for both GPipe and 1F1B is identical. Both schedules suffer from the same overhead of (Number of Stages - 1) bubble steps required to fill and drain the pipe. If you have 8 micro-batches and 4 stages (GPUs), the bubble size is 4 - 1 = 3, and the theoretical efficiency for both schedules is 8 / (8 + 3) ≈ 73%.

So if the theoretical efficiency is the same, why is 1F1B considered superior and used as the industry standard? The answer lies in a critical, practical advantage that the formula doesn't capture: memory efficiency.

The key difference between the two schedules is in how long a GPU must hold onto the intermediate activations it computes during the forward pass. These activations are needed for the backward pass, so they cannot be discarded immediately.

In the GPipe schedule, a GPU must compute and store the activations for all micro-batches during the long, uninterrupted forward phase. The peak memory required is proportional to the total number of micro-batches.

In the 1F1B schedule, a GPU processes the forward pass for a micro-batch _i_, stores its activation, but then relatively soon after, it processes the backward pass for _i_ and can immediately discard that stored activation.

This "first-in, first-out" behavior means the peak memory required at any given moment is significantly lower in a 1F1B schedule. By reducing the peak memory footprint for activations, 1F1B allows us to use a much larger number of micro-batches within the same VRAM budget.

And as we saw in the efficiency formula, increasing the number of micro-batches is the single most effective way to reduce the relative impact of the bubble and push the pipeline's efficiency closer to 100%. Therefore, while GPipe and 1F1B have the same theoretical efficiency for a fixed number of micro-batches, 1F1B's superior memory management allows us to use a much larger number of micro-batches in practice, leading to vastly higher real-world GPU utilization. This is why it is the de facto standard for all modern, large-scale training frameworks.

The 1F1B schedule solves the bubble problem for standard Transformers. However, the introduction of Mixture-of-Experts (MoE) layers, as we saw in chapter 4, brings back a massive communication bottleneck. The All-to-All communication required to shuffle tokens between experts on different GPUs can be even more time-consuming than the pipeline bubble itself. Simply running a 1F1B schedule isn't enough; this new, heavy communication step will leave the GPUs idle once again, waiting for data to be exchanged between stages.

This is the challenge that the DeepSeek team addressed with their training infrastructure. They asked: Can we design a pipeline schedule that not only interleaves forward and backward passes like 1F1B, but also hides the expensive MoE communication at the same time? The answer is yes, and the solution is a highly optimized, bidirectional schedule they named DualPipe Parallelism.

### 6.5.5 The DeepSeek innovation: DualPipe Parallelism

The 1F1B schedule eliminates the forward-backward pipeline bubble for ordinary Transformer layers. However, when Mixture-of-Experts (MoE) layers are introduced, the story becomes significantly more complicated. MoE layers rely on routing different tokens to different experts, and those experts are rarely co-located on the same GPU. As a result, each MoE layer introduces a large cross-GPU data exchange step, where tokens must be dynamically reorganized and sent to the devices that host their selected experts.

This token shuffling step is expensive. For very large MoE models such as DeepSeek-V3, the cost of this communication can exceed the pipeline bubble that 1F1B originally solved. In other words, even if the pipeline itself is perfectly balanced, the GPUs still experience substantial idle time because they are waiting for MoE communication to finish. A vanilla 1F1B schedule cannot hide this cost; something more advanced is required to overlap MoE communication with useful computation and preserve throughput.

The core innovation that enables DualPipe is a concept we'll call pipeline folding. In a standard pipeline, the model's layers are distributed sequentially, like a loaf of bread sliced from one end to the other. If you have 8 GPUs, GPU 0 gets the first slice of layers, GPU 1 gets the second, and so on.

DualPipe rearranges this assignment. A single GPU becomes responsible for two non-contiguous stages: one from the beginning of the model and one from the end. It redistributes stages so each GPU handles one early and one late stage. For an 8-stage pipeline running on 4 physical GPUs (or "ranks"), the mapping would be:

* Rank 0 (GPU 0): Holds Stage 0 (e.g., layers 1-10) and Stage 7 (layers 71-80).
* Rank 1 (GPU 1): Holds Stage 1 (layers 11-20) and Stage 6 (layers 61-70).
* Rank 2 (GPU 2): Holds Stage 2 (layers 21-30) and Stage 5 (layers 51-60).
* Rank 3 (GPU 3): Holds Stage 3 (layers 31-40) and Stage 4 (layers 41-50).

This folded layout is the key that unlocks more advanced scheduling possibilities. While the DeepSeek paper describes a true bidirectional DualPipe schedule that runs two data streams at once, we will focus on a more common and intuitive variant they also open-sourced: DualPipeV. This is also known as the "down-and-up" or "V-shape" schedule.

### 6.5.6 The DualPipe schedule: Hiding MoE communication

The key innovation in DualPipe is not just its bidirectional scheduling but its fine-grained computation-communication overlap. Where 1F1B interleaves entire forward and backward passes, DualPipe breaks these passes down into their even smaller constituent parts—Attention, MLP (Expert), and the expensive All-to-All communication—and meticulously rearranges them.

![Title: dimensions:[378x63] - Description: 069.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/06__image011.png)
*Figure 6.8 The DualPipe schedule with fine-grained overlap. This diagram shows how forward, backward, and overlapped tasks are scheduled to keep the GPUs constantly busy and hide communication latency.*

As illustrated in figure 6.8, the schedule is incredibly complex. The key takeaway is that the scheduler is designed to ensure that while a communication-heavy operation (like the All-to-All for MoE) is happening for one micro-batch, a compute-heavy operation (like the Expert or Attention) is happening for another. For example, a "Backward for weights" computation (blue) might be scheduled right next to a "Forward" computation (orange), allowing the GPU's compute cores and its networking hardware to work in parallel. The bi-color "Overlapped forward & Backward" blocks represent this perfectly, where the forward pass of one micro-batch is happening simultaneously with the backward pass of another.

This fine-grained overlap is what allows DualPipe to effectively hide the massive All-to-All communication cost, solving the primary bottleneck of training large MoE models at scale. While this bidirectional schedule is powerful, an even more practical variant called DualPipeV offers similar benefits with a simpler data flow.

### 6.5.7 A more practical implementation: The DualPipeV schedule

While the full bidirectional DualPipe schedule is powerful, it can be complex to manage as it requires feeding two independent data streams into the pipeline. A more common and intuitive variant, which DeepSeek also provides, is called DualPipeV. This approach, also known as the "down-and-up" or "V-shape" schedule, achieves a similar reduction in the pipeline bubble while using a single, continuous stream of micro-batches.

In the DualPipeV schedule, a micro-batch travels "down" one side of the folded pipeline and then "up" the other side, forming a distinct "V" shape. Let's trace this with our 8-stage model running on 4 GPUs:

* The "Downstroke": A micro-batch starts at Device 0 (Stage 0), is passed to Device 1 (Stage 1), then Device 2 (Stage 2), and finally arrives at Device 3 (Stage 3).
* The "Turnaround": At the "bottom" of the V, Device 3 immediately feeds the output of Stage 3 into the input of Stage 4 (which it also holds). The pipeline "turns the corner" on a single device.
* The "Upstroke": The data now flows in reverse, from Device 3 (Stage 4) to Device 2 (Stage 5), then to Device 1 (Stage 6), and finally completing its forward pass at Device 0 (Stage 7).

The backward pass follows the same V-shape path in reverse. By interleaving the forward and backward passes of different micro-batches, this schedule also dramatically reduces the pipeline bubble, just like 1F1B, but does so on half the number of physical devices.

![Title: dimensions:[378x41] - Description: 0610.png](https://drek4537l1klr.cloudfront.net/dandekar/v-3/Figures/06__image012.png)
*Figure 6.9 The V-Shape Schedule of DualPipeV. Micro-batches flow down through the first half of the stages and then up through the second half, minimizing the bubble while using only N/2 GPUs for an N-stage pipeline.*

As shown in figure 6.9, the DualPipeV schedule retains the core benefit of an interleaved 1F1B-style execution, drastically reducing the bubble compared to the naive GPipe schedule. Its primary advantage is hardware efficiency: it allows a full N-stage pipeline to be run on only N/2 physical GPUs.

This, however, comes with the critical trade-off we identified earlier. Because each GPU in this folded pipeline holds two stages' worth of layers, it must also store 2x the number of parameters compared to a standard 1F1B pipeline. This is a deliberate engineering choice: you trade an increase in VRAM usage for parameters to gain a massive reduction in the pipeline bubble and, in the case of MoE models, a much better ability to hide communication overhead. For a system like DeepSeek-V3, where other techniques (like FP8 and Expert Parallelism) already reduce the per-GPU parameter memory, this trade-off is highly favorable.

### 6.5.8 The trade-offs

Each technique addressing parallelism is a solution to a specific bottleneck, and each comes with its own set of trade-offs.

* Gradient Accumulation: Addresses the batch size memory limit by trading computation time for memory.
* Data Parallelism (DDP): Solves the training time bottleneck by using more hardware, but requires the entire model to fit on each GPU.
* Pipeline Parallelism (GPipe): Solves the model size bottleneck by splitting layers across GPUs, but its naive schedule suffers from massive GPU idle time (the bubble).
* 1F1B Schedule: Solves the bubble problem with an interleaved schedule, but its efficiency is severely hampered by the All-to-All communication required for MoE models.
* DualPipe / DualPipeV: Solves the MoE communication bottleneck by using a folded, V-shape schedule that enables fine-grained overlap. This comes at the cost of requiring 2x the parameter memory per GPU.

This progression shows a clear pattern of engineering: a solution to one bottleneck often reveals another. The DeepSeek architecture, with its combination of Expert Parallelism and a DualPipe-style schedule, represents a sophisticated balance of these trade-offs, purpose-built to train massive MoE models with remarkable efficiency.

With our understanding of the complete training pipeline now established, we can turn our attention to the final stage of our journey: Post-training. While pre-training gives a model its foundational knowledge, the post-training phase is where it is refined, aligned with human preferences, and taught specialized skills. In the next chapter, we will explore the key post-training techniques used by DeepSeek, including Supervised Fine-Tuning and Reinforcement Learning.

## 6.6 Summary

* Building a complete training pipeline involves preparing a dataset, defining the model architecture, writing a training script to optimize the model, and a sampling script to generate text.
* Gradient Accumulation is a technique that simulates a large, effective batch size on memory-constrained hardware by processing smaller mini-batches sequentially and accumulating their gradients before performing a single weight update.
* Data Parallelism accelerates training by replicating the entire model on multiple GPUs, allowing each device to process a different shard of the data batch in parallel.
* The combination of Data Parallelism and Gradient Accumulation is the foundational strategy for scaling training, solving both the time and memory bottlenecks for models that can fit on a single GPU.
* Pipeline Parallelism is a model parallelism technique that solves the "model-is-too-big" problem by splitting the model's layers into stages and distributing them across multiple GPUs.
* The naive GPipe schedule is inefficient due to the "pipeline bubble," a period of significant GPU idle time caused by waiting for the pipeline to fill and drain.
* The 1F1B interleaved schedule drastically reduces the pipeline bubble by alternating forward and backward passes, but its efficiency is hampered in MoE models by the All-to-All communication bottleneck.
* DualPipe Parallelism, DeepSeek's innovation, uses a "folded" pipeline layout and fine-grained scheduling to overlap computation with MoE communication, enabling highly efficient training for massive MoE models at the cost of increased parameter memory per GPU.
